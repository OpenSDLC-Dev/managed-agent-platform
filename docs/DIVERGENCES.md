# DIVERGENCES.md — wire divergences & unconfirmed inferences

The single registry of (a) **deliberate divergences** from the Anthropic Claude
Managed Agents reference and (b) **inferences** about reference behavior made where
neither the SDK types nor the `ant` CLI can reveal it — each to be confirmed against a
real managed-agents recording. The verifier's wire-compat rung resolves intentional
mismatches against this file; an intentional mismatch missing here is a defect. New
divergences land here in the same PR that introduces them.

Format: **surface** — behavior. *Evidence.* Three sections: architecture/compatibility notes (not mismatches), CONFIRMED deliberate divergences, and INFERRED assumptions (each cross-linked to the issue tracking its confirmation).

## Architecture & compatibility notes (not divergences)

Facts that read like divergences but are not mismatches with the managed-agents wire —
recorded here so the allowlist stays honest.

- **Session resource (all session endpoints) — end-user binding** — Sessions carry no user_id field at all. Scoping keys are org/workspace/project (reserved, single-tenant defaults in v1); end-user-to-session ownership is an application-layer concern via session metadata and the audit-only created_by. This is a deliberate architectural divergence (explicitly from adk-go's AppName+UserID pattern; the platform stays user-agnostic). *Evidence: CLAUDE.md 'Non-negotiable design principles' #5; docs/HISTORY.md § "`internal/store` — Postgres schema + migrations" (sessions schema: 'audit-only created_by, no user_id'); CLAUDE.md standing product decisions*
- **GET /v1/sessions/{id}/events/stream — SSE preview delta frames** — Streaming preview deltas use type 'content_delta', never the Messages API's 'content_block_delta'. event_start/event_delta frames are stream-only, never persisted, and reconcile under the buffered event's pre-allocated id. *Evidence: CLAUDE.md 'Wire-compatibility rules'; docs/HISTORY.md § "`internal/events` + events API — append-only log, send/list, SSE stream (slice 3)" (events/preview.go: 'delta type is literally content_delta, NOT the Messages API's content_block_delta')*

- **Session ID prefix parsing (all endpoints accepting a session id)** — Canonical prefix is sesn_; the alternate session_ form is accepted on input, mirroring the reference's dual form. *Evidence: CLAUDE.md 'Wire-compatibility rules'; docs/HISTORY.md § "`internal/domain` — Anthropic-native core types" (domain/id.go 'accepts the alternate session_ form on input'); docs/HISTORY.md § "`internal/api` + `cmd/controlplane` — wire-compatible control-plane CRUD"*
- **All endpoints — anthropic-version / anthropic-beta headers and ?beta=true** — anthropic-version and anthropic-beta request headers are accepted and ignored; ?beta=true is accepted where the reference honors it. No version negotiation is performed. *Evidence: CLAUDE.md 'Wire-compatibility rules'; docs/HISTORY.md § "`internal/api` + `cmd/controlplane` — wire-compatible control-plane CRUD" (api/server.go: '?beta=true and anthropic-* headers accepted and ignored')*
- **Model backend (brain -> provider) — pluggable, config-driven providers** — The model backend is constructed purely from config (protocol anthropic|openai, model, base_url, api_key, optional headers) and routed via a model_providers registry. base_url is required for both adapters — there is no silent api.anthropic.com fallback — and an OpenAI-compatible gateway is a first-class backend. Additive platform surface (deliberate divergence goal: OTel built in). *Evidence: CLAUDE.md principle #3; docs/HISTORY.md § "`internal/events` + events API — append-only log, send/list, SSE stream (slice 3)" (events/span.go), docs/HISTORY.md § "BYOC worker — `traceparent` propagation across the process boundary (slice 8, PR C2b-2)" (PR C2b-2, one OTel trace across the process boundary)*

The wire shape is satisfied; the functionality is deferred. *Evidence: docs/HISTORY.md § "`internal/store` — Postgres schema + migrations" (wire-drift note, resolved by slice 2); reference shape from anthropic-sdk-go betasession.go*
- **Environment scope field** — scope is accepted and rendered only as the constant "organization"; "account" is rejected with a clear error (single-tenant v1). *Evidence: docs/HISTORY.md § "`internal/store` — Postgres schema + migrations", docs/HISTORY.md § "`internal/api` + `cmd/controlplane` — wire-compatible control-plane CRUD" ('scope rendered as the constant "organization"'), docs/HISTORY.md § "`internal/api` + `cmd/controlplane` — wire-compatible control-plane CRUD" (deliberate v1 rejections)*

- **Anthropic prebuilt skills — operator-imported, date-versioned** — The reference hosts its prebuilt skill catalog itself (short-name ids `docx`/`pdf`/`pptx`/`xlsx`, date-based versions); a self-hosted platform has no Anthropic-run catalog to reach, so the controlplane's run-once `-import-anthropic-skills` mode provisions them instead: the operator points it at a local checkout of github.com/anthropics/skills, each directory is validated exactly like an upload, and rows land as `source='anthropic'` with the SKILL.md name as the id and the checkout's last commit date (`YYYYMMDD`, `-import-version` overrides) as the version — this platform mints those versions; the reference's own version values for the same skills will differ. `display_title` is likewise set to the short-name id (`xlsx`, not a marketing name) — the reference's actual `display_title` for these skills is unrecorded and may be human-readable. Anthropic skills are not API-manageable here (`POST …/versions` on one is a 400, and so is `DELETE` of the skill or a version — an accidental delete must not empty a catalog only the importer can refill), matching the reference's server-managed stance. *Evidence: docs/plan/06_skills.md design decision 4 + slice 3; cmd/controlplane (runImport), internal/api/skillsimport.go; managed-agents/skills docs (anthropic skill ids and date versions)* *Tracked: #54.*

- **Skills materialization — sentinel idempotence instead of the reference's re-extract-every-item** — The reference worker re-downloads and re-extracts every skill on every work item and removes `{workdir}/skills` when the item finishes, because its workdir is host-shared across sessions; this platform's sandboxes are per-session, so both halves instead write a `.materialized` sentinel recording the resolved `{skill_id: version}` set (only — the landing directory is deliberately **not** stored) and skip rewriting only when the marker's set is an exact bijection with the freshly resolved set **and** every skill's directory still carries its SKILL.md. Because the workdir is agent-writable the marker bytes are never trusted for anything load-bearing: the probe directory is recomputed each pass from trusted metadata (the executor's DB row / the worker's version GET), never from the marker, so a tool call that rewrites the marker cannot redirect the presence probe at a decoy directory, and — via the exact marker↔resolved-set bijection (equal length, each id once with its version) — cannot use a forged, duplicated, or zero-value entry to mask a skill that is absent from its directory; a removed SKILL.md is healed on the next pass. What the skip does **not** defend against is content-level tampering behind a present SKILL.md, since the probe tests presence, not content — three residuals persist until the resolved set changes: an in-place content edit of any materialized file; a partial deletion that removes a non-SKILL.md member but leaves SKILL.md (the presence probe checks the root SKILL.md as a tractable proxy for the tree, not every member); and — because the landing directory is the skill *name*, shared across a skill's versions — an agent that forges the marker's version to the newly-resolved one while leaving an older version's files in place, suppressing that one in-session upgrade. All three are the same tampering class (an agent that controls the sandbox filesystem can always present chosen bytes under `skills/<name>/`), none grants reach beyond the agent's own single-tenant session, and closing them would mean abandoning the skip to re-extract every pass as the reference does; all are accepted. The reference's re-extract clobbers such edits every item. A rewrite overwrites in place without clearing the skill's directory first (the sandbox seam has no delete primitive), so a file dropped between versions can linger as an orphan beside the new version's complete set — accepted; the reference clears because its host workdir is shared across sessions, which ours never is. The archive is read from object storage / the download stream under a compressed-size byte cap (`skills.ReadArchive`, set above the 30 MB upload limit but far below the gigabyte decompressed ceiling) into a buffer whose capacity is hard-clamped at that cap, so a corrupt or oversized stored object is refused before it can OOM either half and never provokes a gigabyte-scale allocation; extraction's own decompressed guard then bounds what a valid archive expands to. Extraction accepts zip only: the reference's tar fallbacks are dead code against a platform that stores and serves canonical zips. Three further harness-behavior deltas, all defensive and unreachable for archives this platform itself validated: duplicate skill references dedupe first-wins (the reference re-extracts every entry, effectively last-wins); a version name unusable as a path segment falls straight back to the skill id (the reference tries `filepath.Base` first); and extraction refuses traversal/backslash/relative segments outright where the reference absorbs them at the extraction root. One limitation is shared with the reference, not a divergence: because both name the landing directory after the version's name, two distinct skills whose names resolve to the same directory would overwrite each other under `{workdir}/skills/{name}/` — accepted as reference-inherited. *Evidence: anthropic-sdk-go tools/agenttoolset/skills.go (unconditional RemoveAll+extract, Cleanup), skillarchive.go (tar paths); internal/skills/extract.go, internal/executor/skills.go, internal/worker/skills.go* *Tracked: #54.*

- **K8s sandbox backend — deadline and limited-networking fidelity vs Docker** — Two documented necessity divergences from the Docker backend's guarantees: (1) the exec-deadline verdict reads state on the sandbox's own filesystem (Kubernetes has no out-of-band exec handle), so a malicious command can tamper with it in both directions — overwriting the pid file after killing its watchdog to hide an overrun, or writing the watchdog's kill mark and contriving a SIGKILL to claim a timeout it never hit (the mark is what classifies a punctual kill here, since the liveness probe is itself an exec and answers a round trip late — #95, #110). Out of scope for single-tenant v1: honest runaways forge nothing, every term of the verdict only ever *adds* a timeout so a forged mark cannot withdraw one, and a forged timeout mislabels only the tenant's own tool call. Docker keeps no such in-pod state — its probe is a daemon call the sandbox cannot reach — and deliberately still does not; (2) the limited-networking route flush is not equivalent to NetworkMode none on every cluster (raw AF_PACKET sockets can reach the segment; only the main IPv4 table is inspected, so policy-routing CNIs and IPv6 need the reserved egress proxy). *Evidence: docs/HISTORY.md § "Kubernetes sandbox provider (slice 9)"*

- **POST /v1/files — per-organization storage quota not enforced** — The public docs cap uploads at 500 MB per file (enforced, 413) *and* 500 GB per organization; the per-file cap is honored but the per-org quota is deliberately not — a self-hosted operator owns its own disk, and imposing a fixed 500 GB ceiling on a single-tenant deployment would be arbitrary. The per-file 500 MB cap and the documented filename validation (1–255 chars, no `<>:"|?*\/` or control chars) are enforced. *Evidence: docs/plan/08_files.md decision 4; platform.claude.com/docs/en/build-with-claude/files ("File storage and limits"); internal/api/filesupload.go maxFileBytes* *Tracked: #55.*

## Inferences to confirm (INFERRED)

- **POST /v1/skills (+ …/versions) — upload-form detection and rejection surface** — The zip form is detected by magic bytes on a single files[] part (PK\x03\x04); a files[] part without a Content-Disposition filename is rejected 400 (whether the reference tolerates or names such parts "anonymous_file" is unrecorded); flat basenames, mixed top-level directories, and every frontmatter/size violation reject with this platform's own 400 messages — the reference's exact upload-rejection statuses and error strings were never observed (no credentials; the plan's recording checklist covers them). - **Request bodies, path IDs, and query parameters — U+0000 (and any unstorable byte) rejected before it binds** — U+0000 is a well-formed JSON escape — and, percent-decoded from `%00`, a real byte in a path or query value — that Postgres can store in neither a jsonb value nor a text column, so the only alternative to rejecting it is the 500 (`SQLSTATE 22021`/`22P05`) it becomes at bind time; it is rejected in the shape an absent or unknown value already carries, on three surfaces. (1) **Request bodies and inbound event payloads → 400:** a U+0000 anywhere in a JSON object body, key or value, is rejected by `decodeObject` on every endpoint that parses one — not only the metadata-accepting ones (agent / environment / session create and update, and the work-item metadata patch) the guard originally covered — as `internal/events` already rejects it in an inbound payload; this also covers *delete* keys, turning a previously-200 silent no-op (the Go-side merges for agents/environments/sessions dropped an unstorable key harmlessly while the identical patch 500'd against the work endpoint's SQL-side merge) into a 400, the one genuinely wire-observable choice, taken because preserving an asymmetry is not preserving a contract. (2) **Path IDs → 404, id-shaped query filters → 400:** a server-minted id is a known prefix plus a Crockford-base32 token, so anything else — a wrong prefix/alphabet or an unstorable byte — cannot name a stored row, and `domain.ID.Valid` rejects it on shape before the bind, as the 404 an unknown id gets on a path (agents / environments / sessions / events, the work `{work_id}`, and the worker session-read auth lane) and as a 400 on an id-shaped query filter (`agent_id`, the `page` cursor's decoded id).
