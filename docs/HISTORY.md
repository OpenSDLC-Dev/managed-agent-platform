## Slice-8 acceptance — real `ant beta:worker` end to end (run 2026-07-16) — ✅ passed

The plan's slice-8 acceptance, deferred until the local stack existed, has now been run and **passed**.

**Setup.** The docker-compose stack (controlplane + brain + Postgres, no executor — the worker replaces it for self-hosted), the brain pointed at a real Anthropic-protocol endpoint (a MiniMax gateway, from `.env`). A self-hosted environment, an agent carrying the built-in `agent_toolset` (bash), and a session — all created by the **real `ant` CLI** (built from the local checkout) driving the control plane over `--base-url`. An environment key was seeded directly into the DB (the issuance primitive `EnsureEnvironmentKey` has no operator surface yet — see Deferred below).

**Flow exercised.** A `user.message` asking the model to run `echo hello-from-byoc-worker` → the brain called MiniMax, which emitted `agent.tool_use` (`bash`) → because the agent toolset defaults to `always_allow` and the environment is self-hosted, the brain enqueued a self-hosted `tool_exec` work item → a real **`ant beta:worker poll`** (also the local-checkout binary, authenticated with the seeded environment key) polled the environment's queue, claimed the item, reconciled the session by listing its events, ran `bash` in its in-process runner, and posted the `user.tool_result` (`hello-from-byoc-worker\n`) → the brain resumed, produced the final `agent.message` quoting the output, and the session went `idle`. The full event log confirmed the round-trip (`user.message` → `agent.tool_use` → `user.tool_result` → `agent.message` → `session.status_idle`).

**Bug found and fixed (this PR).** The worker's reconcile step — the SDK `SessionToolRunner` listing the session's events, `GET …/events?limit=1000` — was rejected `400` by our shared list cap of 100 (poll and ack had already succeeded), so the worker could never read the outstanding `agent.tool_use` and no tool ran. The reference's general list convention is 1-to-1000 (documented on most SDK list params; the agents list is the documented exception at "maximum 100"), and the worker requests exactly 1000, while the SDK's event-list param documents no explicit cap — so the events endpoint gets `maxEventLimit = 1000` (a compatible upper bound, not a proven reference cap; some cap is needed since an unbounded limit is a query-cost risk) while the other lists keep 100. With the fix the acceptance passes end to end.

**Deferred follow-up.** There is no operator-facing way to **issue** an environment key: `EnsureEnvironmentKey` exists and is tested but is wired to nothing (no endpoint, no CLI, no bootstrap seed). The reference mints these off-wire (its console), so a self-hosted operator needs an equivalent primitive. Tracked for a follow-up iteration (design choice pending: an off-wire admin command vs. a management endpoint).

### `internal/telemetry` — OTel init + W3C trace-context propagation

Uses `go.opentelemetry.io/otel` (+ OTLP/gRPC exporters) — the first external dependency in the module.

| File | Contents |
|---|---|
| `telemetry.go` | `Config` (`ServiceName` / `Endpoint` / `Insecure` / `SampleRatio`) + `Init`: installs the global W3C propagator; with an endpoint configured, installs OTLP/gRPC-exporting tracer + meter + logger providers (`ParentBased(TraceIDRatioBased)` sampler, `service.name` resource). Empty endpoint = fully offline no-op. Returns a flush-at-exit shutdown func. |
| `logs.go` | The slog → OTLP bridge (added 2026-07-17). A configured endpoint points `slog.SetDefault` at a fan-out handler: a `TextHandler` on the console, plus `otelslog` on the logger provider. The fan-out's `Enabled` answers for itself at `bridgeLevel` (Info — the floor slog's own default handler already had) rather than asking its branches, because the OTLP branch has no floor to ask about: `otelslog` has no level option, and its `Enabled` delegates to `sdk/log`'s `BatchProcessor.Enabled`, which is unconditionally true. So adding an endpoint changes where records go, never which records exist. The provider is passed to `otelslog` directly rather than through `otel/log/global`: the global would add a process-wide variable and a second way for two `Init` calls to disagree, and buy nothing. |
| `propagation.go` | `Inject` / `Extract` — W3C `traceparent`/`tracestate` over any `map[string]string` carrier (HTTP request/response headers and a work item's stored trace context both flatten to this shape). Fixed propagator, works without `Init`. |

**The log bridge's one sharp edge**, and why `Init` restores the standard library's `log` writer and flags after installing the handler. `slog.SetDefault` does not only set the slog default: for any handler that is not slog's own `*defaultHandler` it also runs `log.SetOutput(&handlerWriter{...})`, rerouting the whole standard `log` package into that handler. OTel, meanwhile, reports its own export failures with `log.Print` whenever no error-handler delegate is set (`otel/internal/errorhandler`: `ErrDelegator.Handle`), which is our case. Connected, those two close a circuit: an export fails → OTel `log.Print`s it → the line enters the slog handler → the bridge enqueues it as a log record → exporting *that* fails → … Measured against a traces-only collector (Jaeger's shape, and the compose observability profile's own default), **one** ordinary log line produced 2 error lines within 2s and 5 within 8s, still climbing; with the restore, exactly one. Wrapping the handler slog already had is the other tempting shape and is worse — `defaultHandler` writes *through* `log.Output`, so the two deadlock on `log`'s mutex, which is what the `*defaultHandler` type check in `SetDefault` exists to prevent. A `TextHandler` owns its writer and has no such edge; its cost is the console format changing when an endpoint is set, which CHANGELOG declares.

**Test coverage:** contract tests drive an in-process fake OTLP/gRPC collector and assert what actually leaves the process: exported span names, `service.name` resource attribute, exported metrics, exported log records (body, severity, slog attrs), and that `SampleRatio` is honored. Plus traceparent inject/extract round-trip (IDs, sampled flag, remote flag, tracestate) and config validation. The bridge's own tests pin the parts that fail silently: a record emitted under a span carries that span's trace and span id; one emitted without a span still exports rather than being dropped; the console keeps every record; a `Debug` record reaches neither the console nor the collector, while `Info` still reaches both (the level floor the fan-out has to supply, since the OTLP branch has none); a disabled or failed `Init` leaves logging untouched; and — the regression the file exists for — a `log.Print` after `Init` reaches its own writer and **not** the bridge. `span.*` domain-event emission from these spans lands in slice 3.

**Review hardening (same PR, from the dual-review pass):** `created_at` is taken under the session lock via `clock_timestamp()` — the column default `now()` freezes at BEGIN, which would let a lock-waiting transaction write a higher seq with an earlier timestamp and silently break `created_at[gt]` watermark polling; batches insert as one multi-row statement (one round trip under the lock); `\u0000` (valid JSON, unstorable in jsonb) is rejected with a clean 400 instead of surfacing as a 500; `{"type":"text","text":null}` no longer slips through validation; the events cursor binds its sort direction (a follow-up without `?order=` keeps walking the same way, a contradicting one is a 400); the SSE wake path drains preview frames first (an `event_start` can never trail its own buffered event), reads backlog in bounded batches, emits the protocol `error` frame on mid-stream failure, and backstops a lost `session.deleted` broadcast with a ping-time existence check so streams on deleted sessions always terminate; a dropped `event_delta` poisons the rest of its preview so partial text is a clean prefix, never an interior hole; the LISTEN retry loop backs off; span `End` appends the wire event before closing the OTel span and marks the span errored if the append fails; the per-stream preview tracker is capped.

**Slice-3 wire/scope decisions (documented divergences):** the stream is a live tail from connect time — no history replay, no `Last-Event-ID` (the reference client parses neither; reconnecting clients seed via list). `tool_use_id`/`custom_tool_use_id` references were initially not cross-checked on send; slice 5 tightened this — a tool result must reference an existing, kind-matching, not-yet-answered tool use or the send is a 400 (the log is append-only, so one bad reference would poison every future replay). **Known cost, unresolved against the reference:** a worker that loses the HTTP response and re-delivers an identical result gets that 400 rather than a benign success, so re-delivery is not idempotent. Neither the SDK types nor the `ant` CLI reveals whether the real backend rejects or accepts-and-dedups a duplicate reference; settling it needs a recording against a real managed-agents endpoint. Until then we take the strict reading — a rejected send is recoverable, a poisoned session is not. `user.define_outcome` is rejected (outcome surface deferred; no `outc_` prefix in v1). A non-null `session_thread_id` is rejected (threads deferred). `seq` is internal only — never a wire field; cursors carry it opaquely.

**Test coverage:** events package contract tests against Dockerized Postgres — concurrent-append seq integrity (8×25 single session, gap/duplicate-free), cross-pool NOTIFY fan-out, listener kill via `pg_terminate_backend` → reconnect + heal-wake, garbage NOTIFY payloads survived, preview reconciliation (buffered event supersedes deltas under the preview id), JSON-escape-aware chunk reassembly, same-source span emission (one exported OTel span + linked start/end events per request). API contract tests over real HTTP — field-exact echo shapes per inbound type, ~30-case validation sweep (batch atomicity included), cursor walk, SSE framing parsed off the live socket, delta opt-in vs plain subscriber, ping keepalive, `session.deleted` stream termination, corrupt-row 500s.

#### OpenAI-compatible adapter (`internal/provider/openai`)

The second provider protocol, deferred from slice 4 and now landed, completing the v1 requirement that the model backend point at **either** an Anthropic-protocol endpoint **or** an internal OpenAI-compatible gateway. It is registered in `cmd/brain`'s provider registry under `"openai"` alongside `"anthropic"`, so a `model_providers` route with `protocol: openai` resolves to it — no other wiring changes.

Where the Anthropic adapter is near-zero-conversion, this is the platform's **lossy conversion boundary**, kept in one package and tested hard (CLAUDE.md testing conventions). Request conversion (Anthropic-native → OpenAI Chat Completions): the `system` string is prepended as a `system` message; a user turn's `text` blocks become `content` and its `tool_result` blocks fan out into one `tool` role message each (keyed by `tool_call_id`, preceding any user text so they answer the prior assistant); an assistant turn's `text` becomes `content` and its `tool_use` blocks become `tool_calls` (the object `input` is serialized to OpenAI's JSON-**string** `arguments`); Anthropic tool defs become OpenAI function tools (`input_schema`→`parameters`). The request always sets `stream: true` + `stream_options.include_usage: true`; `max_tokens` is omitted when zero so the endpoint applies its own default. Stream translation (OpenAI SSE → chunks): `delta.content`→`text_delta`; a safety `delta.refusal` is surfaced as `text_delta` too (else it would vanish into an empty turn); `delta.tool_calls` accumulate by index and flush as complete `tool_use` chunks at completion (OpenAI has no per-tool-call stop event); the trailing usage frame→`ModelUsage` on the `done` chunk, with `prompt_tokens_details.cached_tokens` split out of `InputTokens` into `CacheReadInputTokens` (matching the Anthropic usage shape); the deprecated single-`function_call` streaming format is rejected loudly (the endpoint must emit `tool_calls`) rather than silently losing the call. **`stop_reason` is `tool_use` whenever the stream carried any tool call** — driven by tool presence, not by `finish_reason` — because some OpenAI-compatible servers (vLLM, some function-calling modes) end a tool turn with `finish_reason: stop`/`length`; honoring that verbatim would emit a `tool_use` event the brain never runs (`tool_use` is the one stop reason it acts on), dangling the turn and, for `length`, poisoning replay with an unanswered tool use. For a turn with no tool calls, `finish_reason` maps to the Anthropic vocabulary (`length`→`max_tokens`, everything else→`end_turn`; the brain flattens every non-`tool_use` stop reason to `end_turn` in v1 anyway, so the distinction is telemetry-only). Completion: a `[DONE]` terminator completes the turn (even from a minimal server that never set `finish_reason`); a body that ends with **neither** a `finish_reason` nor `[DONE]` is a **truncated turn and surfaces as a stream error**, as does a mid-stream `error` frame delivered under HTTP 200 (its message is surfaced, not swallowed as a generic truncation). `base_url` is the API root (the adapter appends `/v1/chat/completions`, matching the anthropic adapter's convention — set it to e.g. `https://api.openai.com`, not `.../v1`). **Documented lossy gaps** (dropped by design, not silently): Anthropic thinking / `redacted_thinking` blocks have no Chat Completions equivalent and are dropped; a `tool_result`'s `is_error` boolean is dropped (OpenAI's `tool` message has no error field — but the error text the platform embeds in the result content is still forwarded, so the model sees the failure, only the flag is lost); image blocks are not yet mapped and an unsupported block fails loudly.

**Test coverage:** contract tests against a fake Chat Completions `httptest` server — a full text+tool round-trip (request assertions: `Authorization: Bearer`, extra header passthrough, model/max_tokens/stream/stream_options, the system+user+assistant+tool message shape, JSON-string tool arguments, function-tool conversion; chunk assertions: text deltas, accumulated tool_use, done with stop_reason + usage), the finish-reason mapping table, string-form content, config validation, upstream 401, and a truncated stream; plus a lossy-paths suite (tool_result block arrays, empty tool_result, assistant-tool-use-only omitting content, empty tool input→`{}`, thinking-block drop, `is_error` forwarding, max_tokens omission, unsupported-block/empty-content/bad-tool-schema errors, no-arg streamed tool call, malformed SSE frame). The dual code review (Codex + the Claude review workflow) hardened the stream translation, each fix pinned by a test: the **tool_use-forcing invariant** (a tool call ending with `finish_reason: stop`/`length`/`[DONE]` still yields `stop_reason: tool_use`), `[DONE]`-without-`finish_reason` and finish-then-EOF completion, mid-stream error-frame surfacing, non-text `tool_result` failing loudly, safety-refusal preservation, cached-token splitting, and legacy-`function_call` rejection; `stream.Close()` drains only a normally-completed body (an early/errored close skips the drain so a hung endpoint can't block the brain's lease-holding defer). The same env-gated integration test as the anthropic adapter (`TestIntegrationRealEndpoint`, gated on `MODEL_PROTOCOL=openai`), skipping cleanly when unconfigured and honoring `-short`.

**Deferred (tracked follow-up):** both provider adapters currently use standalone contract tests; extracting a **shared provider contract harness** (protocol-agnostic invariants — cancellation, close, chunk termination — run against both `anthropic` and `openai`, as `sandboxtest` does for the sandbox backends) is a cross-adapter refactor of its own, not bolted onto this slice. Flagged by Codex; the anthropic adapter predates the convention too, so this closes a gap for both.

### `internal/brain` + `internal/queue` + state machine — the orchestration loop (slice 5)

The platform converses end-to-end. Verified with the real `ant` CLI against the local stack (controlplane + brain + Dockerized Postgres) driving the **real Anthropic-protocol endpoint from `.env`**: `user.message` flips the session to running, a brain claims the turn, replays the log, and the CLI decodes the full turn from both the list and the live SSE stream — `session.status_running`, `span.model_request_start`, `event_start`/`event_delta` previews, buffered `agent.message`, `span.model_request_end` with usage, `session.status_idle` `end_turn` — with session `usage` folded on the resource.

| Piece | Contents |
|---|---|
| `queue/queue.go` | The internal work queue over the existing `work_items` table (`FOR UPDATE SKIP LOCKED`). `Enqueue` is idempotent per (session, kind) while a live item exists (partial unique index, migration `0003`); `Claim` leases the oldest queued item and reclaims expired-active ones (flagged so the brain surfaces recovery); `Extend`/`Complete`/`Requeue` carry the lease expiry as an ownership proof (the reference work API's `expected_last_heartbeat` shape) — a claimant that lost its lease gets `ErrLeaseLost`, never silently finishes a reassigned item. |
| `events` `AppendWith`/`AppendInTx` | Atomic session-state side effects under the append's session row lock: `SetStatus` (resource column and status event can never disagree), `AddUsage` (fold a turn into `sessions.usage`), `MarkProcessedThrough` (stamp consumed inbound events), `Then` (join work enqueue to the same commit). `AppendInTx` lets the API decide the batch under the lock. |
| `api` state machine | `POST /events` is one transaction (`FOR UPDATE OF s`): `user.message` on an idle session → running + `session.status_running` + model_turn enqueue; a tool result while running → next model_turn **only when it completes the set** (every tool use answered — the Messages API rejects a partial replay, so parallel tool calls wait for their last result), no status event (awaiting a tool is still `running`); everything else appends only. Tool results are validated against the log before anything commits: an unknown, kind-mismatched, or already-answered reference is a 400. The response echoes only the posted events. Session updates emit `session.updated` with only the changed fields (title / non-empty metadata / agent snapshot), compared semantically — stored jsonb never byte-matches a fresh marshal. |
| `brain/brain.go` | Claim → replay → generate → **commit the whole turn atomically**: the emitted events (`agent.message`, tool intents), `span.model_request_end`, the status change, the usage fold, the processed watermark, and the work item's fate are ONE transaction under the session row lock, with the queue's lease proof inside it. That single commit is both the liveness guarantee (API triggers serialize on the same lock — a tool result posted mid-settle either sees the live item, suppressed, or the completed one, enqueued; never the gap) and the integrity guarantee (a brain that lost its claim rolls the whole turn back — the log never carries a loser's half-turn, whose duplicate tool intents could never all be answered and would poison every future replay). `tool_use` suspends and completes the item (the intents commit in that same transaction, so nothing can have answered them yet — there is never anything to chain there); `end_turn` idles unless pending input (a mid-turn `user.message` **or** a suppressed tool result) chains the turn by **requeueing its own item**; failures append `session.error` (`model_request_failed_error`) and idle with `retries_exhausted` — unless input is pending, which chains a fresh turn instead of stranding it. The error's `retry_status` follows that decision, not the code path: `exhausted` (terminal) when the session idles, `retrying` when a chained turn is about to run, so a client that stops reading on a terminal error never abandons a session that is still producing events. Both settlements — success and failure — go through one `settle` helper, so the chain-or-idle contract cannot drift between them. Errors are classified: provider/model and deterministic input failures fail the turn visibly; brain-side infra failures (database, lost lease) abandon the turn to lease expiry with nothing committed and nothing on the wire. A lease keeper goroutine re-extends the lease at TTL/3 during streaming (long time-to-first-token can outlast any inter-chunk extension); each renewal is bounded by the lease it is racing (`TTL − TTL/3`, a duration so the deadline never depends on the database and this process agreeing on the clock), so a stalled database cannot hang the turn behind an unreturnable `Extend`, and a renewal only counts as a lost lease once it has actually let the lease lapse. Losing it abandons the turn: nothing has committed, and the item is reclaimed cleanly rather than risking two brains on one session. Reclaimed items surface `session.status_rescheduled` + `status_running` with the lease asserted in the same transaction; the staleness check (status/archived, under the session lock) runs first, so a reclaim of finished work never flips an idle session back to running. |
| `brain/replay.go` | The log IS the conversation: role-run merging, tool_result blocks sorted ahead of text within a user turn, `tool_use` blocks rebuilt under their **event ids** (the provider-side tool id is discarded at emission; result events reference the event id), string content normalized, `system.message` text appended to the system prompt. Custom tools become real tool definitions; `agent_toolset` expansion waits for the executor (slice 6), `mcp_toolset` for the MCP client. |
| `brain/stream.go` | Provider chunks → wire: `agent.message` preview opened at the first **non-empty** text delta (provider block index → content entry index), `agent.thinking` preview per block (start-only, buffered event under the preview id), tool_use collected, the buffered `agent.message` lands **before** `span.model_request_end` (the SDK accumulator closes previews at span end). Empty text deltas are skipped before anything is allocated, so a block that never produces text takes no content index — the preview's delta indices and the stored content array can never disagree, and no preview is opened for an event that will never land. A `tool_use` whose input is not a JSON object (truncated, or a bare string/array/number) fails the turn visibly rather than reaching the append-only log. Database failures mid-stream are marked infra — never reported as model failures. |
| `provider/config.go` + `cmd/brain` | `LoadRoutes` reads the `model_providers` JSON file (`model` / `protocol` / `base_url` / `upstream_model` / `api_key` or `api_key_env`; unknown keys rejected). `cmd/brain`: `DATABASE_URL` + `MODEL_PROVIDERS_PATH` + lease/poll tunables + OTel. |

**Slice-6 decisions (documented divergences, all deliberate):**
- **No `Attach`.** The plan's `SandboxProvider` had `Provision` + `Attach(sandboxID)`. `Provision` is instead idempotent per session — it adopts the session's running container — which is the only thing an executor ever needed `Attach` for, and it spares us persisting a sandbox id nothing else would read.
- **`Glob`/`Grep` are not on the interface.** The plan listed them on `Sandbox`. They are pure functions of `Exec` and the file primitives, so they belong once in the toolset layer rather than re-implemented by every backend. `Checkpoint` is likewise absent: the plan marks it 后续, and a seam is an interface boundary, not a method every backend must stub.
- **`limited` networking fails closed.** The wire's `limited` means "only `allowed_hosts`", which needs the egress proxy the plan reserves. Until that exists a `limited` sandbox gets `NetworkMode: none` — no route out at all. Stricter than the wire contract, never laxer; `allowed_hosts` is accepted and not yet honored.
- **A deadline is enforced twice, and only the outside one is a guarantee.** Docker has no API to kill a running exec, so the killing has to happen inside the container: a wrapper `exec`s the command — so the command *becomes* the exec's own process — after forking a watchdog that, at the deadline, `kill -9`s the command's **process group** (`set -m` makes it a group leader). But that watchdog is a process the command can find and kill. So `Exec` never lets the container decide. It stops waiting `Timeout + 2s` after the deadline (`killGrace`), and it decides the verdict from outside. The watchdog still earns its keep — an honest command's runaway loop stops burning CPU, and the sandbox learns a real exit code — it is simply never believed.
  **Why the command *becomes* the exec, rather than running under a wrapper shell.** The pid Docker reports for an exec is what `Exec` watches from outside. If that pid is a wrapper and the command is its child, the command can `kill $PPID` to make the watched pid vanish while it runs on — a reviewer (Codex) demonstrated exactly that against the earlier design: kill the watchdog, kill the wrapper, run 2s past a 1s deadline, and the call reports `ExitCode=137 TimedOut=false`. `exec`ing the command closes it structurally: the watched pid *is* the command, and a command cannot kill itself and keep running. (The shared contract test now kills the watchdog **and** the parent a prober might watch, then runs long — it fails against the old wrapper and passes against this one.)
  **The verdict is two probes of one question: was the command's process still alive?** Alive as the deadline arrived, and the exec exits 137? Then the watchdog killed it — a timeout. Still alive once the deadline *and* `overrunSlop` (500 ms) had passed? Then no exit code it later reports can be believed, because on the honest path the watchdog would already have killed it — also a timeout.
  **The guarantee, and its soft edge.** A command cannot outrun its deadline by more than `killGrace` — `Exec` stops waiting on its own clock and reports `TimedOut` whatever the container does; that bound is hard. Detecting an overrun *within* that window is softer, because it rests on `top`, and `top`'s answer reflects when the daemon ran `ps`, not when the probe fired — so a command that overruns, exits inside the grace window, and induces the daemon to run its confirming `ps` only after that exit can have the overrun read as a clean finish (up to `killGrace`, and only under attacker-timed daemon delay). The hard bound holds regardless; what leaks is the sub-`killGrace` overrun *label*. Its real containment is the reserved cgroup limits (`PidsLimit`, `CpuQuota`), which also cap the process-count pressure a command needs to stall `ps` — the probe is the cheap first line, not the wall. `TimedOut` is the authoritative field; `ExitCode` may be the kill's, or the one a command that dodged the kill picked for itself.
  Consequences worth knowing: the wrapper keeps **no state inside the container** (a marker file, the first design, let a command forge a timeout it never hit or erase one it did); the deadline a command must have been alive at is the watchdog's rounded-up one, not the caller's request, so a command that kills itself early is never mislabelled; timeouts round up to whole seconds (`sleep` takes no portable fractional argument); the first probe fires `probeLead` (50 ms) *before* the deadline, because at the deadline a command the watchdog has just killed looks exactly like one that was never running — the price is that a command SIGKILLing itself inside that 50 ms reads as a timeout. The watchdog polls `kill -0` on the command rather than sleeping the whole deadline, so an honest command that finishes early takes its watchdog with it within one poll — no stray `sleep` piles up across a session's thousands of quick commands. Its own stdio goes to `/dev/null`; because the command *is* the exec, its stderr reaches the tool result untouched, and a SIGKILL leaves no shell "…Killed…" line behind to begin with.
- **The probe reads the daemon's process list, because both obvious clocks are wrong.** The exec's **output stream** does not close when the command exits: a process the command backgrounds inherits its stdout, and the daemon holds the stream open for it — measured at ~2s, then force-closed. And the daemon's own `Running` flag on the exec tracks that same stream rather than the process — measured at **2.06s** for a command whose process was gone in **40ms**. Timing either one charges a command for stragglers it never waited for, and `sleep 300 & echo started` under a one-second deadline came back `ExitCode: 0` *and* `TimedOut: true`. What does track the process is `GET /containers/{id}/top`, cross-referenced with the exec's `Pid`: `ps` runs on the daemon's host, needs nothing from the image, and is the one view of the sandbox that the sandbox cannot edit. The same measurement fixes the other end — once `Exec` gives up on a stream it asks for an exit code only when the probes say the command is gone, because a real daemon publishes one 1.7s after such a stream is closed and never publishes one for a command still running.
  Two safety properties fall out of doing it this way. A pid the daemon will not name is **fatal**, not ignored: a zero pid would answer "gone" to every probe and disarm the deadline in silence. And a probe the daemon will not answer counts the command as **still running**, because hiding an overrun breaks the guarantee while mislabelling one costs a tool call. That fail-open direction is deliberate, and the overrun probe is careful about it in one more way: its confirmation runs on a clock the output stream's close cannot stop, because a command that overran and then exited *during* its own confirming `top` would otherwise have that exit — not its overrun — read as the answer, and the overrun erased (a reviewer, Codex, found exactly that: the confirmation rode the stream's cancellation, so a clean exit mid-probe came back `TimedOut=false`). What the fail-open direction costs is paid by an honest command a broken `top` can no longer tell from an overrun: one that finished on time but left a straggler holding its output open past the deadline. It reads as a timeout while `top` is down; a working `top` sees the command's pid already gone and clears it, so this misread costs a tool call, never a hidden overrun — a `top` *outage* fails toward the timeout label. (A command that SIGKILLs itself inside the 50 ms probe lead is a separate, unconditional cost of sampling a lead ahead of the deadline: the pre-deadline probe cannot tell a self-kill in that window from the watchdog's, so it reads as a timeout regardless of `top` — the probe-lead cost noted above, not a `top`-outage cost.) Where `top` can still mislead the other way is not an outage but a late-run `ps` answering as of after the command exited: that is the soft edge in the guarantee above, and the reserved cgroup limits, not this probe, are its containment.
- **What the deadline does *not* do: reclaim the container.** The kill is a process *group*, not a process tree, so a child that calls `setsid` escapes it; and when `Exec` stops waiting it abandons the exec — closing the HTTP stream is not a kill primitive. Either way a process the command left behind keeps running, and a sabotaging command can pin a core, until the session's container is destroyed. Every `Exec` call is bounded, so no session wedges, and no command that outlives its deadline escapes the timeout label. What escapes is a process detached from a command that *finished inside* its deadline — `nohup work & exit 0` is honest by every measure the sandbox has, and the work goes on. The real answer is cgroup limits (`PidsLimit`, `CpuQuota`) at provision time, which belongs with the container hardening below, not with the exec path.
- **A container is adopted only if the platform owns it.** `Provision` names a container after the session, so the name alone is not evidence: a container left by an earlier deployment, or one that collides, can wear it. Both adoption paths (an existing container, and the loser of a create race) check the `dev.opensdlc.managed-agent-platform.session-id` label first. The point is not only isolation — a container's **network mode is fixed when it is created**, so adopting a foreign container is how a `limited` session would quietly acquire a `bridge` container's route out, defeating the fail-closed rule above. This is not a trust boundary against a hostile daemon co-tenant — the label is world-writable and forgeable by anyone with daemon access, who already owns every sandbox on the host — it defends against the accidents, which are the realistic failure on a single-tenant daemon.
- **404s are classified by the daemon's own message, anchored.** The archive endpoints echo the requested path into their missing-path error, so a substring search for "No such container" let a file *named* that turn its own missing-file error into a destroyed-sandbox error. The exec endpoints have a third 404, "No such exec instance", which is a lost exec and not a lost sandbox.
- **`Exec` itself is stateless per call.** Each `Exec` is its own `docker exec` of `/bin/bash -c`, so `cd` does not survive between calls at this layer. The reference's `bash` tool *does* persist state (`restart: true` resets it); that persistence is built one layer up, in `internal/sandbox/shell` (below), as a pure function of this stateless `Exec` plus the file primitives — deliberately, so the deadline `Exec` enforces from outside the container is inherited verbatim by every command.
- **A command that backgrounds a process without redirecting its output still costs ~2s.** The command's *result* is correct and its deadline is judged on its own life, but the call does not return until the daemon force-closes the stream the straggler is holding. `docker exec` behaves the same way; `cmd >/dev/null 2>&1 &` returns immediately. Cutting the read short as soon as the probes see the command exit is possible and was not done: it would drop output the command itself wrote and the daemon had not yet flushed, to save two seconds on a tool call the agent chose to shape this way. If it ever matters, it belongs behind the toolset's `bash` tool.
- **Container hardening is not done.** No capability drops, no read-only rootfs, no non-root user, and no resource limits — `PidsLimit` and `CpuQuota` are what actually contain a process that escaped its deadline's kill (see above), and neither is set. `HostConfig.Init` is set (the watchdog leaves short-lived orphans). The plan's shared-responsibility hardening is its own item.

**Test coverage:** the contract suite runs against a real Docker daemon (a missing daemon is a hard failure, not a skip — as with `pgtest`), using `debian:stable-slim`: the official `bash` image does *not* have `/bin/bash` (its bash is in `/usr/local/bin`), which is exactly the assumption the image contract pins. It asserts the deadline kills the command's children along with it, leaves no watchdog behind, leaks nothing into the tool result's stderr, does not mistake a straggler for the command, and does not read a fast exit 124 — the code GNU `timeout(1)` gives a killed command — as a timeout of its own. The crown-jewel subtest tears down every guard the command can see — a watchdog beside it or below it, and the parent a prober might watch in its place — then holds a marker alive past the deadline, and asserts the command never both outlives its deadline and is reported finished; a third case overruns and then exits clean, leaving no marker to see, so the timeout has to be called from the overrun alone; a fourth mutes its own stdout and stderr and then overruns, because a reviewer argued a command could EOF the output stream early and cancel the probes by closing its own fds — measured against a real daemon, it cannot: the daemon holds the exec stream open until the *process* exits, so closing the container-side fds does not close it, the abandoned-plus-overran path still fires, and the command reads as a timeout. It passes against this backend and *fails* against the earlier fork-a-child wrapper, which is the whole reason it is in the shared suite rather than the Docker tests: a backend that enforces its deadline only inside the sandbox, or that watches a killable wrapper, fails it while passing every other assertion in the file. A scripted fake daemon over `httptest` covers what a real one will not reproduce on demand — and models an exec the way a real one runs it, with the process's life and the output stream's life set independently: image-missing → pull → retry, a lost create race adopting the winner's container, a container that wears the session's name without its label, a pull failure delivered inside a 200 stream, garbled and non-JSON replies, start/exec/inspect failures, an exec the daemon never stops calling running, context cancellation mid-poll, symlink and oversize and truncated archives, a `No such exec instance` 404, a path whose own name is `No such container`, the daemon's system-error stream frame, a command that overran and then exited *during* its own `top` probe with the stream closing mid-request, and the unix-socket transport itself.

Every guard is mutation-tested (disable it → exactly its own test fails). Seven rounds of review plus the mutation pass drove the deadline to where it is; the first six found real defects, all fixed here: a non-container 404 from `WriteFile` reported as `ErrNotFound`; the `/tmp` marker a sandboxed command could write to forge or erase its own timeout; a single-PID kill that left the command's children alive; bash's killed-job announcement landing in the tool result's stderr; a leaked watchdog `sleep` per timed command; `TimedOut` compared against the caller's deadline rather than the watchdog's rounded-up one; a path whose name is `No such container` turning its own missing-file error into a destroyed-sandbox error; a stale exec id read as a destroyed sandbox; the daemon's system-error frame folded into stdout as if the command had printed it; a container adopted by name without checking the ownership label; the overrun confirmation cancelled by the command's own stream close, so a clean exit mid-probe erased the overrun; and the two deadline probes run in sequence, so a first probe stalled on a slow `top` left the overrun probe unreached. The seventh round found a residual left in by design rather than a defect fixed here: `top` reports as of when the daemon ran `ps`, so a command that times a daemon `ps`-stall to land just after its own exit can hide a sub-`killGrace` overrun — accepted and documented (in the guarantee above, and as (7) below), with the reserved cgroup limits as its real containment. An eighth round — a fresh Codex (`gpt-5.5`/`xhigh`) and Claude pass on the final branch — turned up no new fixable bypass in the deadline. Its headline finding, a command muting its own stdout and stderr to EOF the stream and cancel the probes before they fire, was reproduced against a real daemon and *did not hold*: the daemon keeps the exec stream open until the process exits, so closing the container-side fds does not close it, and the muted overrun still reads as a timeout (now pinned by the fourth crown-jewel case). Its second finding is a narrow same-family limitation, not a fix: `execPid` polls on the caller's context before the probes are scheduled (anchored to `start`), so a daemon pathologically slow to report the exec's `pid` — slower than `deadline − probeLead` — can push the pre-deadline probe past the watchdog's kill, and a command the watchdog *did* contain reads `TimedOut=false`. Like the seventh, it is a degraded daemon weakening the sub-`killGrace` *label*, never the hard bound (the command was killed on time) and never a hidden overrun; the reserved cgroup limits are the family's real answer. The round's kept findings were otherwise doc and test hygiene: the restored fast-exit-124 guard above, the README daemon rationale, and a `top`-outage note that had wrongly implied a healthy `top` clears an honest self-kill inside the probe lead — it does not, that cost is unconditional.

Seven came from **the sandboxed command attacking its own deadline**. Six were reproduced and fixed; the seventh was reproduced and accepted as a documented residual. (1) Kill the watchdog, run 4.4s past a 1s deadline, report success — fixed by bounding the wait in `Exec`. (2) Once the bound existed, the narrower trick: kill the watchdog, overrun the deadline, and exit *inside* the grace window, so the bound never fires and the exit code is the command's own (measured **2.06s under a 1s deadline, `ExitCode=0 TimedOut=false`**) — fixed by the two probes. (3) The mirror of it, an *honest* command accused: a backgrounded process holding the exec's stdout made `sleep 30 & echo started` read as a timeout — fixed by timing the command's process, not its stream. (4) With the command running as a child of a wrapper shell, `kill $PPID` made the pid the probe watched vanish while the command ran on (measured **2s past a 1s deadline, `ExitCode=137 TimedOut=false`**, reported by two independent reviewers) — fixed by `exec`ing the command so the watched pid *is* the command. (5) Subtler still, once the probes matched the exec's pid: kill the watchdog, overrun the deadline, and exit *during* the overrun probe's own `top` request — the command's stream closes, `Exec` stops probing, and a confirmation riding that cancellation reads the command as finished and returns `TimedOut=false` for a real overrun (Codex, against the exec-wrapper commit) — fixed by running the overrun confirmation on a clock the stream's close cannot stop. (6) The sibling of (5), found re-reviewing that very fix: with the two probes run in sequence, a *first* probe stalled on a slow `top` was still waiting when the command overran and exited, so the stream's close cancelled the whole wait before the overrun probe was ever reached (Codex again) — fixed by running the two probes on independent clocks, so nothing can keep the overrun instant from being measured. (7) The limit behind (5) and (6): even with the probes independent, `top` reports as of when the daemon ran `ps`, not when the probe fired, so a command that overruns, exits inside the grace window, and induces the daemon to run its confirming `ps` only after that exit hides the overrun — bounded by `killGrace`, and only under attacker-timed daemon delay (Codex, re-reviewing the independent-probes fix). This one is accepted and documented rather than fixed: a robust fix means concurrent rapid polling with its own cost and failure surface, and the plan already reserves cgroup limits as the real containment for a command that abuses the daemon — limits that also cap the process pressure needed to stall `ps`. (1)–(4) were each closed with a contract test that fails against the vulnerable code, binding every backend; the general invariant behind (5) and (6) — overran, then exited clean — is a shared subtest too, but the specific `top`-probe races are ones a shared test cannot stage against a real daemon, so they are pinned by Docker unit tests. A verifier pass asserted (2) did not exist and a reviewer proved it did; a verifier pass certified the fix for (1)–(3) and a reviewer found (4) in it; a verifier pass certified (4) and Codex found (5) beside it; a verifier pass certified (5) and Codex found (6) in the re-review; a verifier pass certified (6) and Codex, re-reviewing that fix, found (7) — the point at which the tool reaches the limit of what `top` can witness, and the residual is documented rather than chased. The pattern is why the deadline is pinned by adversarial tests that fail on the vulnerable code, not by a verdict.

The last round found the defect from the other direction — an honest command **accused** of a timeout. A reviewer reasoned that a backgrounded process holding the exec's stdout would stall `Exec` for the full grace period and make it report `TimedOut` and SIGKILL. Reproduced against a real daemon, the mechanism was wrong (the daemon force-closes such a stream after ~2s) but a worse bug was underneath: `sleep 30 & echo started` under a **one-second** deadline returned `ExitCode: 0` together with `TimedOut: true`. Redirecting the straggler's output made it vanish, which named the cause. The fix retired the stream as a clock. Its replacement was chosen by measurement, not by reading the API: the daemon's `Running` flag turned out to track the stream too (2.06s versus the process's real 40ms), and only `top` tracks the process. The mutation pass separately caught a *test* that passed for the wrong reason — the caller-cancellation case tripped during `execStart` and never reached the branch it claimed to pin.

**Coverage gate:** the CI denominator is now logic packages only — `internal/pgtest` and `internal/sandbox/sandboxtest` are excluded, for the same reason `cmd/` always was. They are packages solely because a test in another package must import them, and their uncovered statements are the assertion branches that fire only when a suite fails. The change is load-bearing, not cosmetic: under the old denominator this slice read **89.78%** and CI was red; under the new one, **91.71%** against the unchanged ≥ 90% bar (with the shell package below, the gate now reads **91.78%**). The categorization is what justifies it, not the number — `sandbox/docker` itself covers 96.0%, and what drags the total under the bar is the contract suite's own `t.Errorf` branches. (Excluding only the new `sandboxtest` would also pass, at 91.29%; `pgtest` goes with it because it is the same kind of package.)

### `internal/sandbox/shell` — the persistent bash tool (slice 6, second part)

The reference's `bash` tool is stateful: `cd`, exported variables, functions, and shell options carry from one call to the next, and `restart: true` wipes them. This package provides that on top of the sandbox's stateless `Exec` + file primitives — no new backend surface. (The toolset's `bash` tool and the executor below now drive it; this section was written when it landed on its own.)

| File | Contents |
|---|---|
| `shell/shell.go` | `Run(ctx, sb, session, id, Request)` — one `bash` tool call. `Request{Command, Restart, Timeout}`, `Result` mirrors `ExecResult` plus `Restarted`. It writes the command to a file, mints a directory for this call's snapshot, `Exec`s the embedded template, and then — only if the call did **not** time out **and** left a complete snapshot (its `done` marker) — points `head` at that directory. The snapshot directory is minted per *call*, not named after the tool id: the executor may retry a tool call under the id it already used, and a retry must not inherit the previous attempt's files, least of all its marker. Restart empties `head` through the sandbox's file API (not a container `rm`, which resolves against the container PATH — a prior call that shadowed `rm` on disk would make the reset exit 0 and reset nothing); its write error, not a command's exit code, gates it — a reset that did not run is not a reset. |
| `shell/template.sh` | The only bash the tool introduces, embedded once as data. Go substitutes the (quoted) state dir, tool id and snapshot dir; the user's command is **not** here — it is delivered as a file and `source`d, so no command bytes ride the argument or a sentinel. A prologue restores the committed snapshot (errexit forced off first, options applied last); the shell is then snapshotted into *this call's own* directory, preserving the command's own exit status, and the save ends by creating the `done` marker that makes the snapshot committable. |

**Why a checkpoint and not a resident shell.** A single always-running bash *cannot* keep both an unforgeable outside-the-container deadline and background-straggler correctness in a root-capable sandbox: with the command running *as* the resident shell, foreground-versus-background is shell-internal state the command can rewrite, and the sandbox's rule (`sandbox.go`) that a backend decides `TimedOut` where the sandboxed command cannot reach the decision forces per-command re-exec. So each call is its own `Exec` process — the deadline from PR A applies to it verbatim and unforgeably — and continuity comes from a snapshot on the container's writable layer, restored by the next call.

**The snapshot is committed from outside, on two conditions, and each was a defect before it was a rule.** The obvious design — have the shell overwrite one checkpoint on its way out — is wrong in two ways a reviewer found and a test now pins. A timeout is not always a SIGKILL: a command that kills the in-container watchdog, overruns, and then exits on its own terms runs its EXIT trap perfectly normally, so it would checkpoint the state of a call the sandbox reported as timed out. And a command the sandbox *abandoned* keeps running, so its EXIT trap can land its checkpoint seconds after `Exec` gave up — on top of whatever a later call had written. Both close the same way: the shell snapshots into a directory named by *this call's* id, and the executor commits that snapshot — by writing its id to `head` — only when `Exec` reported no timeout. A timed-out call's snapshot is never pointed at, and an abandoned command's late write lands in a directory nothing will ever point at.

That is necessary and was not sufficient, which a later verifier pass found and a test now pins. "Did not time out" is not the same as "saved": a command can end its shell without ever reaching the save — `exec` replaces the shell, `kill -9 $$` and the OOM killer end it outright, an EXIT trap of the command's own can exit through itself — and none of those is a timeout. Because the template creates this call's snapshot directory on the way *in*, committing on the deadline alone pointed `head` at an empty directory, which does not merely drop that call's mutations: it moves `head` off the last good snapshot and **destroys every earlier call's state with it**. `exec python train.py` is an ordinary thing for an agent to write, it exits 0 with its output intact, and it silently reset the whole session. So the save now ends by creating a `done` marker — written last, and only if every write succeeded — and the commit requires it. A snapshot that was never finished is never committed, and `head` stays on the last one that was. The marker is a *completeness* check against accidents — `exec`, a SIGKILL, an OOM, a broken `PATH`, a mid-save `ENOSPC` — and **not** a trust boundary. It sits on a filesystem the command owns, so a command can forge it and skip the save, committing an empty snapshot and resetting its own shell; that is no more than it can already do by deleting the state directory, the sandbox is per-session so there is no other session's state in the container to reach, and a test pins that blast radius. It does not reintroduce the at-most-once problem removed below, which is about a *retry* re-running a command, not about a command wrecking its own continuity.

**Gating that marker is harder than it looks, and the obvious spelling is broken.** The save writes under `set -e` in a subshell so that a failing write aborts the rest; the marker is then created only if the subshell succeeded. Written the natural way — `( set -e; …writes… ) && : >"$snap/done"` — it does not work at all: **bash ignores `errexit` inside a compound command on the left-hand side of `&&` or `||`, even an explicit `set -e` within that command.** A write in the middle fails, the writes after it run anyway, the subshell ends 0 on its last one, and the marker is created over a snapshot missing a file — which is the original bug wearing a marker (a Codex reviewer found this; `mkdir "$snap/env"` reproduces it, and the next call reads `LOST`). So the subshell is a command in its own right and its status is read from `$?`. The options file has the same requirement from the other side: it must be captured in the *current* shell before `set +e` (or `set -e` can never persist — see the divergence tests), which puts it outside the subshell, so its failure is recorded in a flag rather than swallowed by `|| true`, and the marker requires both. A snapshot committed without its options silently loses every shell option the session had set.

**The save uses builtins only, and reaches them through `builtin`.** Every word that writes the snapshot — `pwd`, `printf`, `declare`, `compgen`, `alias`, `:` — is a bash builtin. There is no `mv`, and atomicity comes from the `done` marker rather than from temp-file renames, so a command that breaks `PATH` is still snapshotted. This is the hardening the *restore* already had (its unset-diff has to survive a snapshot in which the shell unset `PATH` itself) held to on the way out as well: before it, `export PATH=/nonexistent` made every `mv` in the save unresolvable, and the call took the whole session's state down with it.

Being a builtin is not enough, because the save runs in the *same shell as the command* and **a bash function overrides a builtin of the same name** (a Codex reviewer found this). A command that defines `printf() { return 0; }` — a wrapper, not an attack — made the save write an empty `names` file, report success, and earn its marker; the next call then restored `env` and unset every name `names` did not list, which was all of them, leaving the shell without so much as a `PATH`. So every builtin the save depends on is invoked through `builtin`, which a function cannot intercept; `case`, `while` and `[[` are keywords and need no such guard. (A command that defines a function literally named `builtin` still breaks its own save — that is deliberate self-sabotage, not a wrapper, and it lands in the accepted class above.)

**The restore reads `head` and `cwd` with `$(<file)`, not `cat`.** `$(<file)` is bash's own redirection — no command word, so nothing to shadow — and it was the last external in the restore. `cat` there reads from the *container's* PATH, because the restore runs before any snapshot env is sourced, on a fresh exec; a prior call that dropped a program named `cat` earlier in that PATH (a trojan, or an innocent `bat` symlink, and it persists on disk across execs) would make the read return garbage, the `[[ -n head && -f done ]]` guard fail, the whole restore silently skip, and the next call commit the stripped shell over the last good snapshot. A reviewer found it — the one place the restore still broke its own "every word is a builtin, survives a broken `PATH`" invariant. `$(<file)` closes it.

**The restore's unset-diff never word-splits.** It reads names a line at a time under `IFS=`, rather than splitting `$(compgen -e)`. The env it has just sourced may carry an `IFS` of the command's own, and an exported `IFS=` stops that splitting entirely: the diff collapses to a single nonsense word, nothing is unset, and a secret the shell scrubbed quietly comes back from the container environment on the next call — defeating the `unset` divergence exactly where it matters most (also Codex).

**The template's own trailing lines are parsed before any alias exists.** The same round of review found the sharpest version of the shadowing problem: the restore *sources the snapshot's aliases*, and bash expands aliases when a line is **parsed**, so every template line after the restore was alias-expanded. `alias trap=true` — carried, because `shopt -s expand_aliases` is a shell option and options carry — turned `trap '__map_save' EXIT` into a no-op, and from that call on, every command that ended by calling `exit` silently lost its state for the rest of the session. So everything that runs after the restore now lives in `__map_main`, a function *defined before the restore*: a function body is parsed at definition time, when no alias exists yet. The snapshot's alias table is filtered for `__map_*` the same way exports and functions already were (a bare `alias -p` was the one unfiltered table, which also falsified the "template machinery cannot be reached" claim), and every word inside `__map_main` goes through `builtin` against a snapshotted *function* of the same name — **including `.`**, which is a legal function name, is snapshotted like any other, and once restored made the template source nothing at all: every later call returned exit 0 with empty output until a `restart`. A verifier found that one.

**The restore is the other half of that, and the half where shadowing fails *unsafe*.** The `builtin` hardening went into the save and into `__map_main`; the restore kept its bare words, and the next verifier pass found what that costs. The restore *sources the snapshot's functions*, so from that line on the command's own definitions are live — over the restore's remaining words, and over the words the `aliases` and `opts` files themselves run when they are sourced. A command that defined `set() { :; }` cost the session every shell option it had; `shopt() { :; }` the same plus every alias; `alias() { :; }` the alias table; `.() { :; }` and `[() { return 1; }` all of it. And unlike every other shadowing path here, this one **fails unsafe**: the call still finishes, still earns its marker, still commits — but it snapshots a shell the restore had already stripped, so the loss is baked into the new snapshot and is permanent for the rest of the session. Nothing announces it; `cwd` and the environment still come back. So the restore now reaches its own words through `builtin` (`[[` and `case` are keywords and need nothing), and the options are applied a line at a time through `builtin` rather than sourced, because sourcing runs the file's own `set` and `shopt` words in a shell where those may now be the command's.

**Alias expansion reaches inside an already-parsed function body, through any substitution in it — and that one was catastrophic.** A function body is parsed once, when it is defined; that is the property `__map_main` is built on. But bash **re-parses the body of a command or process substitution every time it runs**. `__map_save`'s name, function and alias loops all read from `< <(builtin compgen …)`, so a carried `alias builtin=true` — aliases carry, and `expand_aliases` is an option, so it carries too — rewrote them to `< <(true compgen …)` *inside the already-parsed body*. Every loop then read nothing: `names`, `env`, `funcs` and `aliases` were all written empty, every write "succeeded", the marker was earned, and the empty snapshot committed. The next call restored that empty `env` and unset every name the empty `names` did not list — which was all of them, **`PATH` included**. The save now switches alias expansion off for its own duration, after capturing the options, so the snapshot still records that the command had it on; and the one place the restore must re-parse — applying the options through `eval` — quotes the word, because a quoted word is never alias-expanded. This is why the guards come in pairs: `builtin` stops a shadowing *function*, and only quoting stops a shadowing *alias*.

**The command's ERR and DEBUG traps are dropped before the save runs.** Traps do not carry, so dropping them costs nothing — and keeping them cost the whole snapshot. Under `set -E` (errtrace) or `set -T` (functrace) the command's trap is inherited into the save's process substitutions, and a trap that prints to **stdout** writes straight into the pipe the name-reading loop is consuming: its text is read as if it were a variable name, `declare -p` on that garbage fails, errexit aborts the save, and the call loses everything it did. `set -E; trap 'echo ...' ERR` is an ordinary script idiom, not sabotage. `__map_main` clears the traps after the command, and `__map_save` clears them again on entry, because a command that exits *through* the EXIT trap never reaches `__map_main`'s line.

**Names the template owns are all `__map_*` — including its locals.** `__map_save`'s exit-status local was called `code`, so a command that exported a variable named `code` got the previous call's exit status back instead of its own value (bash gives a `local` the export attribute of a same-named exported global, and the save's `compgen -e` pass then snapshots the local's value). It is `__map_code` now, and the namespace rule is exceptionless.

**Divergences from a resident shell — all of them, each pinned by a test.**
- The **`jobs` table** does not carry. A backgrounded *process* survives (it keeps running, reachable by pid), but the next call is a new shell with an empty job table.
- **Plain (non-exported) variables** do not carry; exported ones do (including readonly exports, and including multi-line values). Nothing in `declare` separates a user's plain variables from bash's own internals, so the snapshot draws the line at `export`.
- **Traps do not carry**, and an EXIT trap fires at most once, inside the call that installed it. It fires only if the command exits *through* it: a command that installs one and then returns normally has it discarded unfired, because the template clears the trap to run its own save. Either way there is no session-long shell for a trap to fire at the end of. A command that both installs its own EXIT trap and exits through it also skips that call's snapshot — the next divergence says what that costs.
- **A call whose shell never finishes its snapshot keeps the *previous* call's state.** Replacing the shell (`exec`), having it killed outright (`kill -9 $$`, an OOM kill), and exiting through an EXIT trap of one's own all skip the save; the call drops its own mutations and the session carries on from the last complete snapshot. A resident shell would have died with the process instead — this is the fail-safe direction of that divergence, not a reset.
- **A timed-out call's mutations are dropped.** A resident shell would keep the ones made before the kill; a SIGKILL leaves no chance to snapshot them, so dropping them is the only behaviour available *consistently* across both the killed path and the dodged-the-kill-and-overran path.
- **The timeout bounds the whole call** — restore, command, snapshot — not the command alone. The bracket is milliseconds, but a very short timeout pays for it.
- **xtrace (`set -x`) does not carry**, though every other option does. Carried, it would have the restore re-enable it and then trace the template's own machinery — the internal state path and the tool-call id — into every later call's stderr, on a call that never asked for it. The call that runs `set -x` still traces its own prologue; the next call starts clean.
- **A command that shadows the template's own machinery costs its own call, not the session.** Its snapshot is not committed and the session carries on from the last complete one. The template's names are all `__map_*` and excluded from the snapshot; every builtin it depends on — in the restore as well as in the save — is reached through `builtin`, and wherever the word is re-parsed at runtime (a substitution, an `eval`) it is quoted as well, since `builtin` stops a shadowing function but only quoting stops a shadowing alias. Snapshotted names are passed after `--` so a function or alias named like an option cannot make `declare -f`/`alias` dump the whole table past the filter. So this takes deliberate sabotage rather than an ordinary wrapper — with one architectural exception, a function named `builtin` itself, which is unguardable and can reset the command's own session (see the limits below).

**At-most-once is not this package's job, and cannot be.** The first cut kept a per-tool-use result marker in the container so a crashed executor's retry would not re-run a non-idempotent command. Three reviewers took it apart from three directions, and they were right: the marker sits on an **agent-writable** filesystem (a command can forge it, or `rm -rf` the state dir and get itself re-run), and it sits on a **cattle** container that a retry may find reaped and re-provisioned, taking every marker with it. A ledger that is neither trustworthy nor durable is not a ledger. At-most-once belongs to the executor and the work queue, whose store is the event log — see "Next up". What survives here is the property that *is* enforceable from outside the container: the deadline.

**The snapshot is the agent's own shell state, not a security boundary.** A command running as root in the container can rewrite or delete it, and only sabotages its own session by doing so. Every name the template owns is `__map_*`, and `__map_*` names are excluded from the snapshot, so an accidental collision cannot reach across into the next call's machinery — but a determined command can still break its own continuity, and that is fine, because the guarantees that matter are enforced where it cannot reach them.

**Test coverage:** Docker-integration tests against a real `debian:stable-slim` container pin every claim above: cwd/env/function/alias/option persistence, `errexit` persistence specifically (the option a naive save always loses, because `set +o` inside a command substitution reports it *off* — a subshell does not inherit errexit unless `inherit_errexit` is set, and it is not by default), readonly exports, multi-line exported values (a line-oriented filter over `declare -px` cuts them in half and takes every variable declared after them down with it), `unset` of a container-inherited variable staying unset (a fresh exec re-inherits the container's environment, so a scrubbed secret would otherwise silently reappear), a command's own EXIT trap not eating the snapshot, `__map_*` names not leaking across calls, exit-code fidelity, a timeout keeping the session while dropping its own mutations — **on both the SIGKILL path and the killed-the-watchdog-and-overran path** — a fast command not reading as a timeout, the PR-A deadline inherited through the bracket, binary-safe separated-and-capped output, and the jobs-table divergence. Six ways to finish a call *without* a complete snapshot and still not time out are a table subtest asserting each drops its own mutations and leaves every earlier call's state standing: `exec`, `kill -9 $$`, exiting through the command's own EXIT trap, sending the save to a directory it cannot write, and — the two that catch a marker created over an incomplete snapshot — making one file of the save unwritable (`mkdir "$snap/env"`) and making the options file unwritable (`mkdir "$snap/opts"`), which is what a mid-save `ENOSPC` or `EIO` looks like. A broken `PATH` is its own subtest, asserting the builtins-only save carries the change rather than costing the session; so is a program named `cat` dropped into the container PATH not breaking the restore (the restore reads `head`/`cwd` with `$(<file)`, no external), a program named `rm` dropped into the PATH not faking a restart (restart empties `head` through the file API, not a container `rm`), and a carried `set -x` not tracing the template's machinery into a *later* call's stderr (xtrace does not carry); so are a command function shadowing a builtin the save uses (`printf() { return 0; }` — the snapshot must survive *and* the function must carry), an exported `IFS=` not disabling the restore's unset-diff, a carried alias on `trap` or `exit` not hijacking the template's own lines, a carried alias on `__map_main` not being snapshotted at all, a function or alias named like an option (`-p`) not making `declare -f`/`alias` dump the whole table past the `__map_*` filter (which would snapshot the template's own `__map_main` and restore it over the real one), a snapshotted *function* named `.` or `trap` not hijacking the words `__map_main` runs (the `.` one silently turned every later call into a no-op), and — nine cases in one table — a snapshotted function or alias not hijacking the **restore**: `.`, `[`, `eval`, `set`, `shopt` and `alias` as functions, `set`, `shopt` and `builtin` as aliases, each asserted to leave the session's options, alias table, environment and cwd intact. That table is the one that fails unsafe when it fails: before the hardening `set() { :; }` silently and permanently dropped every option the session had, and `alias builtin=true` emptied the whole snapshot through the save's process substitutions and left the next call without a `PATH`. Its probe reaches its own words as `\builtin`, quoted, or the poison would silence the probe rather than the template. Also: a printing ERR or DEBUG trap under errtrace/functrace not eating the call's snapshot — including on the path where the command exits *through* the EXIT trap — the template's own post-command lines not firing the command's DEBUG trap into the tool result, an exported variable named `code` not being clobbered by the save's local, a retry under an already-used tool id getting a fresh snapshot directory, a reset whose `head` write failed failing the call, and the forged-marker blast radius (a command that forges `done` and skips the save resets its own session and leaves a second session in the same container untouched). A fake-sandbox unit suite (no daemon) pins the commit rule from both sides — a complete snapshot commits, a timed-out call does not commit *even a complete one*, and a call that left no marker leaves `head` on the previous snapshot — plus every backend-fault path, including a snapshot probe against a broken container failing the call rather than reading as "nothing was saved". Every guard is mutation-tested, and each mutation fails exactly the test that claims to pin it: dropping the `done` gate fails the no-snapshot cases and the unit test behind them; restoring the `( set -e; … ) && : >done` spelling fails the two incomplete-snapshot cases; putting an external `mv` back into the save fails the broken-`PATH` case; dropping the `builtin` prefixes fails the shadowed-builtin case; reverting the unset-diff to `for … in $(compgen -e)` fails the exported-`IFS` case; moving `__map_main`'s body back to the top level fails the aliased-`trap` case; un-`builtin`-ing `.` or `trap` inside `__map_main` fails the hijacked-word cases; letting `alias -p` snapshot the whole table fails the aliased-`__map_main` case; not clearing the ERR/DEBUG traps in `__map_save` fails the two printing-trap-that-exits cases; clearing only EXIT in `__map_main` fails the DEBUG-trap-noise case; renaming `__map_code` back to `code` fails the clobbered-variable case; dropping the restart `head`-write error check fails the failed-reset case, and reverting the restart from a `head` file-write to a container `rm` fails the shadowed-`rm` case; naming the snapshot directory after the tool id again fails the retry cases; and, in the restore, un-`builtin`-ing `.` or `eval`, putting `[` back in place of `[[`, sourcing the options file instead of applying it a line at a time through `builtin`, dropping the `\builtin` prefix the save writes in front of each alias record, dropping the backslash in front of the eval'd option, and dropping the save's `shopt -u expand_aliases` each fail exactly the restore-hijack case that pins them; and dropping the `--` from the save's `declare -f`/`alias` fails exactly the function/alias option-injection case; restoring `cat` in place of `$(<file)` for either `head` or `cwd` fails the shadowed-`cat` case; and turning xtrace off *after* the opts capture instead of before it (so it carries again) fails the xtrace-does-not-carry case. The package covers 100% of its statements.

**Known limits, accepted for now.**
- **Snapshots and command files accumulate.** Every call writes both and nothing prunes them; `restart` prunes nothing either. The container is per-session and disposable and each snapshot is a few KB, so a long session costs tens of MB and a destroyed sandbox takes it all with it — a garbage collector is not worth its failure modes yet.
- **A `Run` that fails *after* the command ran discards the command's output.** The snapshot probe or the `head` write against a broken container returns the error with an empty `Result`. That is deliberate: the alternative hands the caller a transcript for a call whose state the next call will not see, which invites treating it as committed. Both reviewers raised it; it stays a documented choice rather than a fix.
- **A command that defines a function named `builtin` is unguardable, and a crafted one can reset its own session.** `builtin` is the one word that routes a call around a shadowing function, so a function that *is* `builtin` cannot be routed around — and no keyword can enumerate and serialize the shell, so there is no builtin-free fallback (an *alias* named `builtin` is fully guarded, because the word is quoted wherever it is re-parsed; only a function is not). Two shapes, both deliberate self-sabotage — a wrapper around `printf` is plausible, one around `builtin` is not. Written to return 0 it spins the save (`while IFS= builtin read` never terminates), measured at 100% of a core, and the call times out having committed nothing — its own call only. Written to break a single builtin the save needs (`compgen`) while delegating the rest, it does *not* spin: the save writes an empty snapshot, earns its marker, and commits, resetting the session's `head` to a shell with no state. The blast radius is still the command's OWN session — `head` is per-session, and a second session in the same container is untouched (the same bound as the forged-marker case) — and the ultimate containment is the same the sandbox owes any command that abuses the container: the deadline, and the `PidsLimit`/`CpuQuota` above. Codex found the shape (its specific delegating-`builtin` trigger, aimed at the save's `shopt -u expand_aliases`, does not reproduce — a delegating `builtin` snapshots correctly — but a `compgen`-breaking one does); it is documented rather than fixed because it is not fixable inside a shell whose every builtin the command may shadow.
- **`set -x` still leaks a handful of the template's own trace lines** into the tool result's stderr — but only on the call that ran it. xtrace does not carry (see the divergence): the save turns it off before it captures the options, so it is recorded off and the next call is clean. Within the call that ran `set -x`, `__map_main`'s own lines and the save's first line are traced before that switch-off, and a command that asked for `set -x` gets them; a reviewer found that it used to carry and trace every *later* call too, which is what turning it off before the opts capture fixed.
- **Concurrent `Run` calls on one session race on `head`.** Two in-flight calls snapshot into separate directories and the last to commit wins. Nothing consumes the package yet, so it cannot bite; the executor must serialize a session's bash calls, and that belongs with the executor.
- **A cwd whose name ends in a newline is not round-tripped.** The snapshot stores the cwd with `pwd` and restores it through `$(<file)`; command substitution strips *all* trailing newlines, so a directory whose own name ends in one is restored a byte short and the `cd` lands elsewhere or is skipped. It is a pre-existing property of the substitution (the earlier `$(cat …)` behaved identically), it costs only cwd (exports, functions, aliases, options are unaffected), and a newline-terminated directory name does not occur in real use — reading the value byte-exact to handle it is machinery a hot, safety-critical path does not warrant. Codex raised it.

### `internal/toolset` — the built-in tools (slice 6, third part)

`agent_toolset_20260401`: the six tools the platform runs for the model, executing inside the session's sandbox. (The executor and the brain's `agent_toolset` expansion below now offer these to the model and run them; this section was written when the toolset landed on its own, before a consumer existed.)

| File | Contents |
|---|---|
| `toolset/toolset.go` | `Runner` (sandbox + session + workdir) and `Run(ctx, id, name, input)`. The one line the package draws: a **tool** failure (missing file, bad regex, nonzero exit, a non-regular target, a NUL in a path) is a `Result` with `IsError` — the model reads it and recovers — while a **backend** failure (sandbox gone, daemon unreachable) is an `error` and never a result the model would try to reason about. The whole glob pipeline is **NUL-delimited** (`stat --printf … \0` / `sort -z`), so a matched filename that itself contains a newline stays one record and cannot inject a fabricated path; `pipefail` is on so a broken pipeline (an image whose `stat` lacks `--printf`, a missing tool the up-front `command -v` guard also names) is a reported error rather than a silent "no matches", with the one accepted cost that a match unlinked mid-listing makes the call a retryable error instead of returning the survivors. grep uses the image's GNU grep, PCRE where it has it (a model writes `\d` far more readily than `[0-9]`), ERE where it does not; exit 1 is the answer "no matches", not a failure. |

**Slice-6 toolset decisions (documented divergences from the SDK's `tools/agenttoolset`, all deliberate):**
- **No workdir confinement.** The reference runs the tools on the host, so it has to canonicalize every path and reject escapes — and says outright that its `bash` cannot be confined at all. Here the container **is** the boundary and `bash` runs in it like everything else, so a lexical check the model can walk around with one `bash` call is theatre, not a boundary. Relative paths resolve against the workdir; absolute paths and absolute glob patterns are accepted.
- **The pattern is data, and bash is what guarantees it.** A model-supplied pattern is single-quoted into the script as a *variable*, and expanded unquoted only where a pathname pattern belongs — bash globs an expansion's result but never rescans it for command substitution, so `$(touch /tmp/pwned)/*.go` matches nothing and runs nothing (pinned by a test that fails the moment the value is interpolated raw). `IFS` is emptied so the value is one word: a pattern may contain spaces.
- **One grep, not ripgrep-or-a-walker.** The reference prefers `rg` and falls back to a Go walker, which means two behaviours (rg honours `.gitignore`; the walker does not) decided by what the image happens to ship. One implementation, one behaviour. The toolset layers a stricter image requirement on top of the sandbox's bash-plus-POSIX baseline: **grep and glob need GNU coreutils/grep/findutils** — grep's `--exclude-dir`/`-I`/`-P`, glob's `stat --printf`/`sort -z`/`xargs -0`. That is what any bash-carrying distro image (the v1 default is `debian`) ships; a deliberately busybox-only image is out of contract, and grep/glob report a clear tool error there rather than degrading. A ripgrep-or-walker style fallback is exactly the two-behaviours complexity this rejects.
- **`web_fetch` / `web_search` stay deferred.** They are in the wire's tool-config enum but carry no input schema there and run executor-side against an egress policy that does not exist yet. Enabling one offers the model nothing, and calling it is an error result rather than a tool call that hangs.
- **Interleaving is lost.** The sandbox captures stdout and stderr separately, so a tool result carries stderr after stdout whole rather than interleaved as a terminal would show it. The reference's bash tool runs on a PTY and merges them; matching that would mean giving the sandbox a PTY, which is not worth what it costs.
- **A glob record splits its mtime off on the first space**, and records split on NUL, so a filename may contain spaces *and* newlines and still round-trips as one path. (A filename ending in a newline is the sole exception, via `stat`'s own output shape — the same narrow class as the shell package's newline-terminated cwd.)
- **A tool call's deadline is the model's, clamped.** `timeout_ms` is honoured up to `MaxTimeout` (10 min) and defaults to 2 minutes; the clamp is applied before the millisecond count is scaled, so a value large enough to overflow a `Duration` becomes the maximum rather than an instant deadline. The executor's lease must outlive `MaxTimeout` — that is a constraint on the executor, recorded here because this is where the number lives.

**Known limits, accepted for now (recorded from the dual review):**
- **A write to a path *under* a non-directory surfaces as a backend fault, not a tool error.** `write file_path="/a/regular/file/child"` makes the docker sandbox return `http 500: not a directory`, which carries no sentinel, so `fileFault` routes it as an `error`. The principled fix is typed `ENOTDIR`/`EISDIR` errors from `sandbox.WriteFile` (the docker backend), which is sandbox-layer work; recorded so the executor's fault policy accounts for it. (A write *to* an existing directory is a separate, milder sandbox oddity: the tar put returns nil, so the tool reports a spurious success rather than a fault.)
- **`write`/`edit` are not atomic.** The reference writes to a temp file and renames; `sandbox.WriteFile` extracts a tar onto the target, so a write that fails mid-transfer can leave a truncated file. Within a session it cannot corrupt a concurrent read — the executor serialises a session's tool calls — so the residue is only a failed write's own file. Atomicity, like the typed errors above, belongs in the sandbox backend.
- **`singleQuote` is a third copy** of the POSIX single-quote escape (`sandbox/docker` and `sandbox/shell` have their own). Each is one line and independently tested; a shared home would couple three sandbox-tree packages for a trivial helper, so the duplication stays, as the shell package's own note already accepts for its copy.

**Test coverage:** the six tools run against a real `debian:stable-slim` sandbox (a missing daemon is a hard failure, as with the other suites): bash's stdout/stderr capture, state carrying across calls, restart with and without a command, a nonzero exit, a timeout that reports no exit code, `read`'s ranges (inclusive, EOF, past-the-end, inverted), `write`'s parent creation and overwrite, `edit`'s unique-match rule and `replace_all`, a non-regular (FIFO) `read`/`edit` surfacing as a tool error, glob's doublestar/single-star/dotfile/newest-first/no-matches/limit behaviour and its absolute-pattern-ignores-`path` rule, grep's `path:line:text` output, its PCRE class, its binary and `node_modules` skips, and its cap — plus the injection pair on both search tools (a `$(…)` payload in the pattern runs nothing; a `$(…)` in a *matched filename* is not re-expanded either), a pattern containing a space, and a newline in a matched name not fabricating a second path. A fake-sandbox unit suite pins what a daemon will not produce on demand: every backend fault surfacing as an error rather than a tool result, the file sentinels (including the new not-a-regular-file one) surfacing as tool results rather than errors, a NUL byte in any path/pattern caught as a tool error before it reaches the sandbox, paths resolving against the workdir, the deadline reaching the sandbox on every call (default, honoured, clamped, overflow-clamped), a search timeout, a silent search failure, the 200-path cut, the absolute-pattern root, a grep regex that starts with `/` still rooting at the workdir, the output cap landing on a rune boundary, bash's exit-code/timeout trailer surviving truncation of a huge output, and every tool's malformed-input message. Guards are mutation-tested: dropping `IFS=` fails the space-in-pattern case, interpolating the pattern raw fails all three injection cases, dropping the output cap fails the two truncation cases, dropping the timeout clamp fails the two clamp cases, dropping the not-a-regular-file classification fails both the fake-suite sentinel case and the real-container FIFO case, and reverting bash to append-then-cap fails the trailer-survival case. The package covers 97.7% of its statements; the CI gate reads 92.18% total.

The glob pipeline's failure handling is carried on reasoning rather than a failing-on-the-old-code test: a broken pipeline (a `stat` without `--printf`, a missing tool) and a mid-listing `stat` race are both non-deterministic to stage against a real GNU-coreutils container, so `pipefail`'s "never a silent no-matches" guarantee rests on the happy-path glob tests proving the NUL-delimited pipeline works plus the argument that any masked mid-pipeline failure would otherwise read to the model as an empty directory. A second reviewer (Codex) confirmed the masking behavior a `pipefail`-less pipeline would have had, which is why `pipefail` is kept and the per-file-race-errors-conservatively cost is accepted.

### `internal/executor` + `cmd/executor` + brain `agent_toolset` expansion — the closed loop (slice 6, fourth part)

The *hands* now have a consumer. When the model calls a built-in tool the brain emits `agent.tool_use` and suspends the turn, and an executor runs that tool inside the session's sandbox and answers it — the platform-managed half of the pull protocol (`cloud`), the same protocol a BYOC worker will speak in slice 8. This closes the loop the plan's v1 goal names: `agent.tool_use` → an executor runs the tool in a sandbox → `agent.tool_result` → the brain resumes.

| Piece | Contents |
|---|---|
| `brain/replay.go` | `buildRequest` now expands each tool entry by type: a `custom` tool becomes a client-executed definition (as before), and an `agent_toolset_20260401` entry is expanded through `toolset.Tools` into the built-in definitions the model sees — so the model is finally offered bash/read/write/edit/glob/grep. `classifyTools(agent)` maps each tool name to the event type its use commits under: a custom tool → `agent.custom_tool_use` (client-executed, never enters the queue), a built-in → `agent.tool_use` (platform-executed). |
| `brain/brain.go` | `turnEvents` stamps each model-emitted tool intent with the type `classifyTools` resolved (unknown names default to custom, the safe direction — a client-executed intent never strands a session). When a turn suspends on a **platform** tool use, `commitTurn` enqueues one `tool_exec` item in the same transaction that commits the intents and completes the model_turn item — so the work to run the tools is scheduled atomically with the intents themselves. A turn that suspends only on custom tools enqueues nothing (the client answers those). |
| `executor/executor.go` | The `tool_exec` consumer. `Run` polls the queue; `step` claims the oldest `tool_exec` item (reclaiming an expired lease) and `process` runs it: load the session's egress policy, provision the sandbox (idempotent per session), gather the session's **unanswered** tool uses, run them, and commit the results, the resume, and the item's fate together. The append's `Then` — under the session row lock — enqueues a `model_turn` **only when every tool use is answered** (`events.HasUnansweredToolUse`, the same full-set gate the control plane uses for client results), and completes the item only when every tool actually ran. |
| `executor/toolwork.go` | `unansweredToolUses` diffs the session's `agent.tool_use` events against the `agent.tool_result` events already on the log, oldest first — the work this item must run, and the reclaim ledger. The lease keeper (`keepLease`) renews the claim at TTL/3 while tools run and cancels the work context if it ever loses the lease, mirroring the brain's; the renewed lease is the proof the settling commit uses. |
| `cmd/executor/main.go` | The binary: `DATABASE_URL` + `EXECUTOR_IMAGE`/`EXECUTOR_WORKDIR` + lease/poll tunables + `DOCKER_HOST` + OTel, wiring the Docker sandbox provider into the executor. Disposable "hands" — run as many as needed. |

**The scheduler trap the toolset section flagged is closed by the executor carrying its own resume enqueue.** The turn scheduler only ever sees *inbound* results: `pendingInputTypes` lists `user.message` and the two `user.*_tool_result` types, and every platform-emitted event is stamped `processed_at` at insert — so an `agent.tool_result` the executor appends while a turn is live would be suppressed by the live work item and missed by the settle's pending check. The executor therefore does not rely on the scheduler noticing: its result append carries the `model_turn` enqueue itself (its own `Then`, mirroring the API's client-result trigger), so the resume is scheduled in the same commit as the results. `HasUnansweredToolUse` already counted `agent.tool_result` as an answer; only the scheduling half was missing, and it lives with the appender.

**At-most-once is the executor's, decided here, and it is a re-run bound not a re-run ban.** At-most-once lives in the queue's lease, not in a marker inside the sandbox (which is agent-writable and disposable — the shell package's section explains why a container marker cannot be a ledger). The lease keeps two executors off one session's tools; a crash mid-run lets the lease lapse, and the reclaiming executor re-derives the work from `unansweredToolUses`, so it re-runs **only the tools still unanswered** — a committed result is never re-run. A tool's *result* is therefore exactly-once even though a non-idempotent *command* (a tool that ran but crashed before its result committed) can run more than once across a crash. That residue is inherent to a disposable sandbox and is documented, not solved: the sandbox has no rollback, and the alternative — a durable pre-execution marker — is the container-marker ledger three reviewers already took apart.

**A tool fault is answered; a backend fault is reclaimed.** `runTools` keeps the toolset's own split: a tool that fails *at the tool level* (missing file, nonzero exit, a NUL in a path) still yields an `agent.tool_result` with `is_error` — the model reads it and recovers — so the set completes and the turn resumes. A **backend** fault (sandbox gone, daemon unreachable, a write to a path under a non-directory) stops the set, leaves the tool and everything after it unanswered, and does **not** complete the item: no `model_turn` is enqueued, the results that did run are still appended (so a reclaim skips them), and the item is left live for its lease to lapse and another claim to retry the rest. A per-item fault never crashes the loop — only a fatal claim failure returns up.

**The two constraints the toolset put on the executor are met.** The lease TTL defaults to 15 min, comfortably above `toolset.MaxTimeout` (10 min), so a single long tool cannot let the lease lapse mid-run (and the keeper renews besides). And a session's `bash` calls are serialized without extra machinery: the queue's per-(session, kind) dedup keeps at most one live `tool_exec` per session, its lease keeps a second executor off it, and one executor runs a session's tools sequentially — so two `bash` calls never race on the shell snapshot.

**Test coverage:** executor contract tests against Dockerized Postgres with a fake in-memory sandbox — one tool runs and the result lands referencing its use, the `model_turn` resume is enqueued, and the `tool_exec` completes; parallel tools all answer before a single resume; a tool-level error (a missing-file read) still resumes rather than stranding; a backend fault (sandbox write fails) commits nothing, enqueues no resume, and leaves the item live for reclaim; a provision fault does the same; a reclaim re-runs only the still-unanswered tool and leaves the already-committed one alone; `Run` drains a queued item and stops cleanly on cancel; an empty queue reports no work. The lease keeper is pinned from both sides against a gated tool: a run that outlives TTL/3 has its lease renewed in the background and still commits under the renewed proof, and a lease stolen mid-run makes the keeper's next renewal fail, which cancels the work and aborts the commit (no result, no resume, item left for reclaim). Plus a real-container closed-loop integration test: one `bash` tool driven the whole way through a live Docker sandbox — the brain's suspend shape (a `tool_use` plus one `tool_exec` item), the executor claiming it, running `echo` in the container, appending the result, and scheduling the resume. The brain-side change is covered by the slice-5 brain suite extended: a built-in tool use enqueues a `tool_exec` (where a custom tool use enqueues nothing), and `classifyTools`/`agent_toolset` expansion are pinned in the replay tests. The executor package covers ~84% of its statements (it varies a point or two run to run — the lease-keeper renewal/extend-error branches are timing-dependent; the remainder is defensive error branches — an unreachable `json.Marshal` failure, injected claim/query errors); the CI gate reads **~91.7%** total (no credentials, so the provider integration test skips, as on CI).

**Review hardening (same PR, from the dual review + verifier).** A Codex (`gpt-5.5`/`xhigh`) pass, a Claude multi-agent review, and the verifier converged on a set of correctness gaps, all fixed here:
- **A dead session no longer reclaim-loops.** `archiveSession` stamps `archived_at` on a session of any status and does not drain its work; a session archived while suspended on a tool would have its `tool_exec` item fail every append (`ErrSessionArchived`) and be reclaimed and **re-run forever**, each pass provisioning a container and re-executing the tools. `process` now opens with `sessionForRun`, which loads the session under its row lock and drains (completes) the item if the session is not running or is archived — mirroring the brain's `claimLiveSession`, which guards the same loop. The egress-policy load folds into that one locked read.
- **A tool answered by `user.tool_result` is not re-run.** The canonical answered-set (`events.HasUnansweredToolUse`) counts a self_hosted worker's `user.tool_result` as answering an `agent.tool_use`; `unansweredToolUses` counted only `agent.tool_result`, so the executor would re-run a tool a worker already answered and append a duplicate result. It now counts both.
- **The backend-fault partial commit proves its lease.** A backend fault commits the results that did run (so a reclaim skips them) but completed nothing else, carrying no ownership proof — a claim lost while blocked on the session lock could still commit a result a reclaiming executor also wrote, duplicating it on the append-only log. The fault path now `Assert`s the lease inside the same transaction, like every other state write.
- **The lease keeper covers provisioning.** It now starts *before* `Provision` (an image pull can be slow) and provisioning runs under the kept context, so the lease cannot lapse mid-pull and let a second executor reclaim.
- **File tools share bash's workdir.** `runTools` now builds the `toolset.Runner` with `e.cfg.Workdir`, so a relative path resolves against the same directory bash runs in; before, under a non-default `EXECUTOR_WORKDIR`, the file tools used `/workspace` while bash used the configured dir.
- **An empty tool result is an empty content array, not an empty text block.** A `read` of an empty file yielded `content:[{type:text,text:""}]`; a Messages endpoint rejects an empty text block, and that request is what the brain replays every resume — wedging the session. `toolResultEvent` now emits `[]` for empty output.
- **Faults are logged, and a tiny lease TTL cannot panic.** Per-item faults `slog.Error` (the brain logs turn failures the same way) so a Docker outage is visible rather than a silent stall; the recovery is still the lease's reclaim, so the item waits out its lease as before. A sub-3ns `EXECUTOR_LEASE_TTL` no longer feeds `time.NewTicker(0)`. `Config.BlockOnEmpty` (redundant with `PollInterval`) was removed, and `classifyTools` is skipped on a text-only turn.

**Two review findings are documented rather than fixed, both malformed-config edges the current stack already fails visibly on:**
- **A custom tool named like a built-in yields duplicate tool definitions.** With `agent_toolset` now expanded, an agent that also defines a custom `bash`/`read`/… hands the model two tools of that name; the Anthropic Messages API rejects duplicate tool names with a 400, so the turn fails visibly rather than misrouting (the misroute in `classifyTools`'s name map is unreachable while the provider rejects the request). The proper fix is tool-name-uniqueness validation at agent creation — the same class as the numeric-cap and mcp cross-check validations already deferred in slice 2 — not executor work.
- **The lease keeper is a near-verbatim copy of the brain's.** Both drive `queue.Extend` on a `queue.Item` with the same ticker and bounded-timeout math; a shared `queue`-level keeper is the eventual home, deferred as a focused chore that touches the brain (the same call the shell package's triplicated `singleQuote` makes).

**Deferred to slice 7 / follow-ups (recorded, not built):** nothing destroys a sandbox yet — session termination and orphan reaping by the ownership label belong with the executor's lifecycle and are unbuilt. Container hardening (`PidsLimit`, `CpuQuota` — the real containment for a process that escaped the deadline's kill) is still open. Adoption re-validates the ownership label but not the network mode, so once a caller can vary a session's networking, adoption must reject a container whose mode is not the one now asked for. None of these can bite today: sessions are not yet terminated through this path, and networking is fixed per session.

**The executor was scoped to `cloud` in slice 8 PR A (this note originally deferred it).** Through slice 7 `queue.Claim` filtered only by kind, so the platform-managed executor also picked up a `self_hosted` environment's `tool_exec` work — deferred because the BYOC worker that consumes `self_hosted` work did not exist and filtering to `cloud`-only would have left a `self_hosted` tool call hanging. Slice 8 PR A introduced `Poll` (the wire work API) and, with it, the queue-level split: `Claim(tool_exec)` now serves only `cloud` and `Poll` only `self_hosted`, so a polled item is never also run by the executor (`Claim(model_turn)` stays unscoped — the brain runs on the platform for every environment). A `self_hosted` `tool_exec` item now correctly waits for its worker, which the slice-8 detail block above covers. The defensive `unansweredToolUses` answered-check remains as crash-recovery belt-and-suspenders. (Originally flagged P1 by the Codex GitHub app on PR #15; resolved here.)

### permission policies + the confirmation round-trip (slice 7)

An `always_ask` built-in tool now suspends the turn for one human approval before it runs — the confirmation gate sits exactly where the plan's v1 goal loop puts it, between the brain's suspend and the executor's claim. The gate spans four packages, and it is one atomic change by necessity: the brain's suspend must not land before the API can resolve a confirmation, or an `always_ask` session would hang with no way to unblock it.

| Piece | Contents |
|---|---|
| `domain/agent.go` | `EvaluatedPermission` (`allow` / `ask` / `deny`) — the resolved decision the brain stamps on an `agent.tool_use`. `deny` is reserved: no configurable policy produces it (a denial is expressed as an error result, below), but the wire enum carries it. |
| `toolset/definitions.go` | `Policies(entry)` → each enabled built-in tool's resolved `permission_policy`, keyed by name; a shared `resolveToolset` now backs both `Tools` and `Policies`, so enable resolution (per-tool config > `default_config` > on) and policy resolution (same precedence, default `DefaultAgentToolsetPolicy`) can never disagree about which tools exist. An unknown `permission_policy.type` on an enabled tool is a hard error — a policy the platform cannot evaluate never silently resolves to "run it anyway". |
| `brain/replay.go` + `brain/brain.go` | `classifyPolicies(agent)` resolves the agent-toolset tools' policies (custom tools are client-executed and carry none). `turnEvents` stamps `evaluated_permission` on every platform `agent.tool_use` (`allow` or `ask`) and pre-mints the `sevt_` ids of the `ask` intents so the same ids can name them in the stop_reason. When any intent is `always_ask`, `commitTurn` gates the **whole** turn: it emits `session.status_idle` carrying `stop_reason:{type:"requires_action", event_ids:[…ask ids]}`, sets the session `idle`, completes the model_turn item, and enqueues **no** `tool_exec` — even the `always_allow` tools in that turn wait for confirmation. An all-allow turn takes the slice-6 path unchanged (suspend `running`, enqueue `tool_exec`). |
| `events/toolflow.go` | `ValidateToolConfirmations` rejects a confirmation that does not name an ask-gated, not-yet-confirmed tool use (the append-only-log discipline the tool-result validation already had). `UnconfirmedAskEvents` returns the still-blocking ask ids, treating a validated-but-uninserted batch's confirmations as resolved — the API's input for deciding resume-vs-re-idle. `ToolConfirmationRefs` collects a batch's referenced ids. (Inbound field validation for `user.tool_confirmation` — result `allow`/`deny`, `deny_message` only with `deny`, null thread — already lived in `events/inbound.go` from slice 3.) |
| `api/events.go` | `POST /events` gains the confirmation trigger. On a batch of confirmations against an `idle` session it appends an `agent.tool_result{is_error:true}` answering each **denied** tool (content = the client's `deny_message`, or a default — never an empty text block), then computes the remaining blocking set: still blocked → re-emit `session.status_idle` with the shrunken `event_ids` (status stays `idle`); fully resolved → flip `running` and enqueue the work that finishes the turn — a `tool_exec` when an allowed tool still needs running, or a `model_turn` directly when every gated tool was denied (nothing to run, the brain resumes on the error results). |

**Two wire-schema decisions rest on the plan and inference, both flagged for a recording against a real endpoint:**
- **The agent-toolset default policy is `always_allow` (`DefaultAgentToolsetPolicy`).** The plan states the reference resolves the agent toolset to `always_allow`; the SDK wire types carry no resolved default to corroborate it, and one create-agent example in the SDK resolves a kitchen-sink agent to `always_ask`. The plan is authoritative here and `always_allow` preserves the working slice-6 loop (a bare toolset keeps auto-running), so it is the default — as a single named constant to flip once a real managed-agents endpoint can be recorded.
- **A denial is an `agent.tool_result` with `is_error:true` carrying the `deny_message`.** The reference documents the `user.tool_confirmation` event but not the result a denial produces. The model protocol requires every `tool_use` answered before the turn resumes, so a denied tool must carry *some* result; an error result whose content is the deny message is the natural inference (and what lets the model see and react to the refusal). Recorded as an inference, not observed behaviour.

**Design decision — a mixed turn gates everything, not just its ask tools.** When a turn calls both an `always_ask` and an `always_allow` tool, the platform blocks the allow tool too rather than running it while the ask tool waits. It is simpler (no partial-execution bookkeeping while a confirmation is outstanding) and safer (an agent's plan is often a sequence; running step 2 before a human vetoes step 1 is worse than waiting), and it costs only latency on the uncommon mixed turn. `requires_action.event_ids` still lists only the ask tools — that is what the human must act on — but the resume that runs the allowed tools waits for those confirmations.

**Liveness — a gated turn re-idles rather than chains, and that is deliberate.** Both the brain's suspend and the API's partial re-idle commit under the session row lock with no chain-or-idle check (unlike `end_turn`): a session blocked on human confirmation is genuinely waiting on input, and a `user.message` that arrived mid-turn stays unprocessed and rides the next replay once the gate clears — it is never lost, and the session is not spuriously woken past a pending approval. This mirrors the slice-6 running-suspend, which also relies on replay-on-resume for mid-turn input.

**Test coverage:** toolset resolver tests (bare entry → every tool `always_allow`; `default_config` policy applied to all; per-tool override; disabled tools absent; unknown/malformed policy rejected). Brain suspend tests over real Postgres (an `always_ask` tool idles with `requires_action` naming its tool_use event and enqueues no `tool_exec`, the intent carries `evaluated_permission:"ask"`; a mixed turn gates everything but names only the ask tool, each intent recording its own permission). API state-machine tests (allow → running + `tool_exec`; deny → error result carrying the message + `model_turn`; partial confirm → re-idle with the remaining `event_ids`, then full confirm resumes; mixed allow+deny in one batch; validation 400s — unknown ref, `deny_message` on allow, intra-batch duplicate, non-ask tool, already-confirmed). And two **brain-to-API integration tests** that stitch the halves: a real brain runs an `always_ask` turn and suspends, then a `user.tool_confirmation` POSTed through the API resolves the *exact* id the brain minted into `requires_action` — proving the two sides agree on the event id, for both the allow (resume + `tool_exec`) and deny (error result + `model_turn`) paths.

**Review hardening (same PR, from the dual review + verifier).** A Codex (`gpt-5.5`/`xhigh`) pass, the verifier (opus), and the Claude review (also `gpt-5.5`-class) converged; seven gaps — six correctness, one cleanup — were fixed here, each correctness one with a test that fails on the vulnerable code:
- **A `user.message` no longer bypasses the confirmation gate.** Slice 7 is the first to leave a session *idle* while it still carries an unanswered `agent.tool_use`, and the `user.message → running + model_turn` trigger did not check for that: a message posted while awaiting approval woke a turn whose replay hands the model an assistant `tool_use` with no matching result — a request the Messages API rejects, producing a spurious `session.error`. `POST /events` now computes the session's unconfirmed-ask blocking set up front (`UnconfirmedAskEvents`) and a `user.message` resumes only when it is empty; while gated the message appends and rides the next replay once the gate clears. The confirmation case is checked **first**, so a batch mixing a gate-clearing confirmation with a `user.message` runs the confirmed tool rather than waking on the message past it. (All three reviewers found this independently — the strongest signal in the pass.)
- **A tool result can no longer answer an unconfirmed ask tool.** `ValidateToolResults` accepted a `user.tool_result` for an ask-gated `agent.tool_use` (it only checked kind and already-answered), so a self_hosted client could answer a gated tool before approval — bypassing the human gate and, on a later denial, double-answering the tool use on the append-only log. It now rejects a result for an ask-gated tool that has no confirmation. (Codex.)
- **A malformed toolset is a 400 at agent creation, not a turn-time wedge.** With `Tools` now resolving `permission_policy` (via the shared `resolveToolset`), an agent whose `agent_toolset` carried a bad policy — or a malformed `enabled` — was accepted at create (`parseTools` treated the entry as opaque) and then failed *every* turn when the brain resolved it, an unusable agent with no create-time error. `parseTools` now calls `toolset.Validate` on each `agent_toolset` entry, so the resolver's own check runs at creation. (Claude.)
- **A denied gate no longer provisions a sandbox for nothing.** The confirmation resume chose `tool_exec` vs `model_turn` from `HasUnansweredToolUse`, which counts client-executed `agent.custom_tool_use` as unanswered — so denying an ask tool while a custom tool was still outstanding enqueued a `tool_exec` the executor would claim, provision a container for, and find no built-in to run (a wasted provision; an infinite reclaim if provisioning failed). The resume now enqueues a `tool_exec` only when an allowed *platform* tool is unanswered (`HasUnansweredPlatformToolUse`); when the only remaining work is a custom tool it enqueues nothing and waits for the client's result, and when every tool is answered it resumes the brain — mirroring the non-ask suspend, which never runs an executor for a custom-only turn. (Claude.)
- **Confirmation gating is scoped to `agent.tool_use` only.** `confirmableToolUseTypes` also listed `agent.mcp_tool_use`, but the denial synthesis emits an `agent.tool_result`/`tool_use_id` — the wrong shape for an MCP tool (`agent.mcp_tool_result`/`mcp_tool_use_id`). MCP is not gated in v1 (the brain stamps a policy on nothing else), so the speculative entry was removed; gating MCP is slice-8+ work that must extend the denial synthesis with it. (Codex.)
- **Policy validation is uniformly lazy.** `resolveToolset` validated `default_config.permission_policy` eagerly while per-tool policies were validated only for enabled tools, so a malformed default rejected an agent even when every enabled tool overrode it or the toolset was off. Now a policy is validated only for a tool that actually resolves into the enabled set — a malformed policy on a disabled or overridden-away tool has no effect and is ignored. (Codex.)
- **(cleanup) One `classify` pass replaces the parallel `classifyTools`/`classifyPolicies`.** Both re-ran `resolveToolset` over the same entry each tool-calling turn, and `classifyTools` marshaled every built-in definition back to JSON just to recover its name; `classify` now resolves once and reads names from `toolset.Policies`' keys, so the two maps (name→event type, name→policy) can no longer drift. (Claude.)

The verifier's remaining notes were non-blocking: the new `events/toolflow.go` validators are covered through the `internal/api` confirmation tests (the same convention as the slice-5/6 tool-flow helpers in that file), and an end-to-end `ant`-CLI smoke of the confirmation path needs a live model to non-deterministically call a gated tool — the DB-backed integration tests exercise the same handler and a real `brain.RunOnce`.

**Slice-6 seam closed:** `toolset/definitions.go`'s comment that "`permission_policy` is read by nobody yet; policies … are slice 7" is now false and updated — `Policies` reads it.

### work API — environment-key auth + `/work/poll` (slice 8, first part)

BYOC workers speak a **second auth domain**: `Authorization: Bearer <environment key>`, not the management `x-api-key`. `EnsureEnvironmentKey` registers one live worker credential per environment (hash-only, revoke-others-on-re-mint — the `EnsureAPIKey` shape, scoped by `environment_id` instead of a logical name); `requireEnvironmentKey` resolves a Bearer token to its environment and the poll handler asserts that environment matches the path's `{id}`. The work routes and the management routes share **one mux**, and a top-level `dispatchAuth` picks the scheme by path — Bearer for `/v1/environments/{id}/work…`, `x-api-key` for everything else — so authentication always runs **before** any `ServeMux` path-cleaning or subtree-slash redirect (an unauthenticated request never gets a bare 3xx instead of the 401 wire envelope). `GET …/work/poll` returns the oldest queued `tool_exec` item for the environment as a `BetaSelfHostedWork` whose `data` is a reference to the **session** the worker attaches to (`{id:"session_…",type:"session"}`) — the work item wraps a session, not a tool_use, and results go back to the session events API (`user.tool_result`), never to the work API. `queue.Poll` is a **soft reservation**: unlike the executor's `Claim` (`queued→active` lease), the polled item stays `queued` and a later `ack` transitions it; the reservation is a lease pushed out by `reclaim_older_than_ms` (clamped so an over-large value can't overflow into a past lease), so a worker that dies between poll and ack strands nothing.

**The cloud/self_hosted split lands at the queue here** (its worker-consuming half is PR C). `Claim(tool_exec)` now serves only **cloud** environments — the platform-managed executor is the cloud hands — and `Poll` serves only **self_hosted**; `Claim(model_turn)` is unscoped (the brain runs the model on the platform for every environment). Cloud and self_hosted `tool_exec` queues are therefore mutually exclusive by environment kind, so an item a worker has polled can never also be run by the executor. For that exclusivity to be airtight the routing key must not move under a live work item, so **an environment's kind is immutable after creation**: a config update that would flip `cloud`↔`self_hosted` is rejected (a kind change would re-home a session's compute and re-route its queue mid-flight — see the divergence below). A consequence to hold until PR C: a self_hosted `tool_exec` item has **no consumer yet** (the worker binary is PR C), so a self_hosted session that calls a built-in tool waits — correctly, since a self_hosted environment has no hands until its worker runs. (Before this split the executor claimed every environment's work; the slice-6 note documenting that is now resolved.)

**Slice-8 part-1 decisions (documented divergences/assumptions, each to confirm against a real managed-agents recording):**
- **Environment-key issuance is the platform's own primitive.** The reference mints environment keys in its console; neither the SDK nor the `ant` CLI exposes a wire endpoint that creates one (the CLI worker only *consumes* `--environment-key`). A self-hostable platform must provision them, so `EnsureEnvironmentKey` owns that shape. The *consuming* side (Bearer → environment) stays wire-locked by the real `ant beta:worker`.
- **Empty poll is `200` + `null`.** The reference client reads "no work" from either a nil body or an empty id; `204` is equally plausible server-side. Both are client-compatible; `null` keeps the endpoint's success status uniform at `200`.
- **`block_ms` is accepted but the poll is non-blocking** — it returns immediately, and the reference client already spaces empty polls with a client-side jitter sleep, so the protocol is unchanged (only chattier). True long-poll (hold the request open on a `work_items` NOTIFY) is deferred.
- **Unreached lifecycle timestamps render as `null`.** A queued item has not been acknowledged/started/stopped and has no heartbeat, so `acknowledged_at`/`started_at`/`stop_requested_at`/`stopped_at`/`latest_heartbeat_at` are `null` (the SDK's string-typed, `api:"required"` fields tolerate null). Columns for these arrived with PR B (below), where the transition endpoints populate them.
- **A Bearer key for the wrong environment is `401`**, not `403`/`404` — the credential simply does not authenticate for this environment (no `permission_error` type exists in the envelope; the poller treats any 4xx as a permanent stop regardless). A key value is also bound to one environment for life: re-minting the same value for a different environment is rejected rather than silently re-pointing a live credential.
- **An environment's kind is immutable after creation.** A config update that flips `cloud`↔`self_hosted` is rejected with a `400`. The wire types don't reveal whether the reference forbids this, but a kind change re-homes a session's compute and (here) re-routes its work between the executor and a BYOC worker, so this platform rejects it as a conservative safety divergence — to confirm against a recording. Config updates *within* a kind are unaffected.
- **`traceparent` propagation reaches the BYOC worker via a `/work/poll` response header, and stores in a dedicated column (landed, PR C2b-2).** The plan requires the turn's trace context to reach the worker "through the work item", but the SDK work schema has no such field — so the resolution is a `traceparent`/`tracestate` **response header** on `/work/poll` (the real worker ignores unknown headers; ours reads them via `option.WithResponseInto`). **Divergence from the plan's sketch:** the plan implied riding the work item's `metadata`; we store it in a dedicated `trace_context` `jsonb` column (migration `0005`) and emit the header from *that*, never the wire body. Rationale: the work-item `metadata` namespace is now client-updatable (PR C-meta), so an internal `traceparent` must not pollute it — a separate column keeps it out of the client-facing surface entirely, with no leak risk and no strip step.

**Known limitation (accepted, mirrors `EnsureAPIKey`):** two `EnsureEnvironmentKey` calls for the *same* environment racing under Postgres READ COMMITTED can each insert a different value and neither's revoke-step sees the other's uncommitted row, leaving two live keys briefly (same-environment only — no cross-environment reach). Key minting is a serialized admin action, so this needs concurrent mints for one environment to bite. The clean fix — a partial unique index `(environment_id) WHERE revoked_at IS NULL` plus a revoke-before-insert order — applies equally to the pre-existing `api_keys` rotation and is deferred so both are hardened together rather than diverging.

### work API — the work-item lifecycle: get / ack / heartbeat / stop (slice 8, second part)

A polled item now runs a full state machine to `stopped`. Migration `0004` adds the four lifecycle-timestamp columns (`acknowledged_at`/`started_at`/`stop_requested_at`/`stopped_at`) the part-1 wire shape already rendered as `null`; `queue.Work` gains them plus a shared `workColumns` projection and `scanWork` decoder, and four endpoints drive the transitions:

- **`GET …/work/{work_id}`** returns one item (unknown → `404`).
- **`POST …/work/{work_id}/ack`** advances `queued → starting` and stamps `acknowledged_at`. It is **idempotent** — only the `queued→starting` edge stamps the timestamp, so a re-ack of an already-advanced item returns it unchanged rather than erroring (a worker that retries a lost ack response is safe).
- **`POST …/work/{work_id}/heartbeat`** is the wire's **optimistic-concurrency lease**. The first heartbeat sends `expected_last_heartbeat=NO_HEARTBEAT` to claim a just-acked (`starting`) item → `active`, stamping `started_at`; every later heartbeat echoes the server's prior `last_heartbeat` and extends the lease while `active`. On a **present** item, a value that isn't the row's current `last_heartbeat` — a stale/replayed/superseded one, or a `NO_HEARTBEAT` re-claim — is `412`; a heartbeat on an item that no longer exists (missing/wrong-env/reclaimed-then-deleted) is `404`, so a worker can tell "my value is stale, refetch" from "this item is gone". A heartbeat on an item the control plane has moved to `stopping`/`stopped` still matches (the worker's echo is current) but does **not** extend, and reports the new state, so the worker learns to wind down. `desired_ttl_seconds` (default 30, clamped to 300) sets the effective TTL; the response is `BetaSelfHostedWorkHeartbeatResponse` (`last_heartbeat`/`lease_extended`/`state`/`ttl_seconds`/`type:"work_heartbeat"`).
- **`POST …/work/{work_id}/stop`** takes `{force?:bool}`. A graceful stop (`force:false`/empty body) moves a live (`queued`/`starting`/`active`) item to `stopping` so the worker can wind down; `force:true` escalates any not-yet-stopped item straight to `stopped`. It returns **`200` + the updated `BetaSelfHostedWork`** (like ack/heartbeat); an item already past the requested transition (graceful-stopping a `stopping` item, or stopping a `stopped` one) is `409`, which the reference worker ignores.

All four endpoints (and `Poll`) scope to a **self_hosted `tool_exec`** item: the two other row kinds sharing the `work_items` table are invisible to a worker's environment-key endpoints — `model_turn` is the brain's own queue (acking one would wedge the brain's turn) and a **cloud** environment's `tool_exec` is the platform executor's (force-stopping one would yank it mid-run). A work id outside that scope is `404`.

**Dead-worker reclaim (PR C3) recovers a worker's in-flight item.** `Poll` reclaims two kinds of stranded item. A still-**`queued`** (un-acked) reservation whose window (`reclaim_older_than_ms`) lapsed is re-offered — the reference's documented "reclaim un-ack'd work". AND an item a worker **acked** (`starting`) or **heartbeated** (`active`) and then died on — its **lease** (`lease_expires_at`: the startup lease `Ack` installs on the queued→starting edge, then extended by each heartbeat) has lapsed — is reclaimed: `Poll` resets it to a fresh `queued` reservation (`last_heartbeat`/`acknowledged_at`/`started_at` cleared) so the next worker re-polls, re-acks, and re-claims it with a fresh `NO_HEARTBEAT`. The startup lease matters: without it a `starting` item would carry only the short un-acked poll reservation, and a slow-but-live worker's item could be stolen in the ack → first-heartbeat gap. This mirrors `Claim`'s expired-active reclaim for cloud, and the active-item reclaim keys on the **lapsed lease**, not on `reclaim_older_than_ms` (which stays the un-acked-reservation window, per the wire). The C2a driver re-derives work from the still-unanswered tool uses, so a reclaimed run re-executes only unanswered tools; a revived stale worker learns it lost the item on its next heartbeat (echoed `last_heartbeat` no longer matches → `412`). Wire-settled against the reference: the work item carries **no** generation/version field and `stop` carries **no** ownership proof (body is `{force}` only), so the "lease-guarded stop" option is off the table — the server-internal requeue-in-place reclaim is invisible to the client and the `412`-on-heartbeat is the reclaim signal. **Residual (documented, not a v1 blocker):** a *hung-then-revived* worker could, in the tightest race, complete its tools before its heartbeat `412`s and `stop(force)` the replacement's reclaimed item, re-stranding the session — the identity-blind-stop limitation the reference shares. A truly dead worker never revives, so the plan's kill-worker resilience case is fully covered; the C2b `!lostLease` guard covers the common sub-case; fully closing the tight race needs a fresh work identity per hand-out, a later hardening.

The optimistic-concurrency round-trip is **instant-based, not string-based**: `last_heartbeat` is stored as `timestamptz`; the echoed precondition is parsed (`RFC3339Nano`) and matched as a bound `time.Time` (`= $n`) so an instant, not a text form, is compared — a timezone-representation change can never spuriously mismatch, and a malformed value is a `412` rather than a cast-error `500`. The queue layer owns the state machine (`queue.Ack`/`Heartbeat`/`Stop`/`GetWork` with `ErrWorkNotFound`/`ErrWorkConflict`/`ErrHeartbeatMismatch`); the API layer maps those to `404`/`409`/`412` and is otherwise a thin wire adapter.

**Slice-8 part-2 decisions (documented divergences/assumptions, each to confirm against a real managed-agents recording):**
- **`expected_last_heartbeat` is required — absent → `400`.** The SDK marks it optional (`omitzero`) and the `ant` reference command leaves the flag optional, but the automated worker (the only real consumer) always sends it — `NO_HEARTBEAT` to claim, then the echoed value — and there is no sound default without it: claim-vs-extend is exactly what the precondition selects, and a heartbeat with no precondition would defeat the optimistic-concurrency the mechanism exists for. Requiring it never rejects a real worker; only a hand-rolled client omitting it sees the `400`.
- **Graceful vs force map to `stopping` vs `stopped`.** The wire exposes only `{force?}` and the `state` enum (`…/stopping/stopped`); it does not spell out the two-step wind-down, so this platform reads `force:false` as "ask the worker to stop" (`stopping`) and `force:true` as "stop now" (`stopped`), with a heartbeat during `stopping` as the worker's acknowledgement channel. To confirm against a recording.
- **Stop returns `200` + the `BetaSelfHostedWork`, not `204`.** The SDK types `Work.Stop` → `*BetaSelfHostedWork`, and driving the pinned SDK against a `204`/empty-body server makes its typed decoder error (`expected destination type … that is not 'application/json'`) — so a `204` is not wire-compatible even though the reference worker happens to swallow the error. Stop therefore echoes the updated item like ack/heartbeat. A conflicting stop is `409 invalid_request_error` (the envelope has no dedicated conflict type), which the reference worker ignores.
- **The metadata update (`POST …/work/{work_id}`) and the `list`/`stats` reporting endpoints were deferred here (PR B).** None was on the BYOC worker's critical path (poll→ack→heartbeat→stop), so each followed as its own small PR; at this point `POST …/work/{work_id}` (update) was an unimplemented method on a known path and answered `405` via the method-less fallback. **(Update: `list` landed in PR C-list, the metadata update in PR C-meta below — now the live update handler, so a `POST …/work/{work_id}` no longer hits the `405` fallback; only `stats` remains.)**

### work API — environment-key auth on a session's worker-facing routes (slice 8, PR C1)

The BYOC worker's server-side prerequisite: a worker drives its session over the **same** `Authorization: Bearer <environment key>` it polls work with (the reference environment key authorizes both the work-poll calls and the session-level calls). The worker-facing session routes — `GET`/`POST /v1/sessions/{id}/events`, `GET …/events/stream`, **and the `GET /v1/sessions/{id}` read** — are now **dual-auth**: a worker's `Bearer` key or the management `x-api-key`. The exact set is what the reference `ant beta:worker` uses the environment key for: the session-events tool runner (`Beta.Sessions.Events.NewToolRunner` → stream/list/send) and the session read `Beta.Sessions.Get` its `SetupSkills` performs. Only the **read** verb of the bare `/v1/sessions/{id}` joins the set (a `GET`); its mutating verbs do not.

Everything runs in the same top-level `dispatchAuth` that already splits work vs management paths, so auth still runs **before** any `ServeMux` redirect. Two correctness points settled here:

- **Classification uses the escaped path** (`URL.EscapedPath()`, split only on real `/`), so the worker lanes are strictly no broader than what `ServeMux` routes: a route is recognized only when its literal segments (`events`/`stream`/`work`/`poll`) appear unencoded, exactly as every real client sends them. Classifying on the percent-decoded `URL.Path` let a `%2F` forge an extra segment the router does not see — `GET /v1/sessions/{id}%2Fevents` decodes to an events path (Bearer-authable) yet the router matches it as `/v1/sessions/{id}` CRUD, so an environment key would reach a management-only handler. The reverse case is **fail-closed** and driverless: a request that percent-encodes a literal segment (e.g. `/%65vents`) is not taken as a worker route and falls to management auth (`401`), never an over-authorization. One `splitSession` splitter feeds both the auth-lane predicates and the middleware's ownership id (on the decoded path, so it equals the handler's `PathValue`).
- **The dual-auth lane picks the environment key only when a `Bearer` is present AND no `x-api-key` is.** The reference client deletes `x-api-key` before attaching the environment `Bearer` (the server rejects both at once), so `x-api-key` present unambiguously means a management caller; keying on `Bearer` presence alone would let a stray `Bearer` header knock a valid `x-api-key` caller off management auth.

Scope is enforced in a middleware (`requireEnvironmentKeyForSession`), not scattered across the handlers: after resolving the key it looks up the target session's `environment_id` and requires it to equal the key's environment. For a given id, a session in **another** environment and a session that **does not exist** take the identical branch and return the same `404` (status, type, message) — a worker probing an id can neither reach another environment's sessions nor learn they exist. **Mutating** session CRUD (`POST`/`DELETE /v1/sessions/{id}`, `…/archive`, and the collection routes) is deliberately **not** in the dual-auth set: it stays management-only, so a `Bearer`-only request to it falls through to management auth and is rejected for the missing `x-api-key`. The Bearer-resolution logic (`resolveEnvironmentKey`) is shared by both worker middlewares (work API and session routes) so the credential rules live in one place.

### work API — the work-items list (slice 8, PR C-list)

`GET /v1/environments/{id}/work` — the read-only reporting list deferred in PR B. It returns the environment's work items as `BetaSelfHostedWork` objects in the platform's standard `{data, next_page}` envelope (opaque forward cursor keyed on `(created_at, id)` DESC, `?limit` validated to 1–100 — a value outside the range is a `400`, the same as every other list here), and is scoped exactly like the rest of the work API — self_hosted `tool_exec` items only, so a worker's list never shows the brain's `model_turn` rows or another environment's work. `queue.ListWork` owns the SQL + scope; the handler owns the cursor. Environment-key auth (a wrong-env key or the management `x-api-key` is `401`); a write method such as `POST` is `405` (`HEAD` is served as `GET`, per the router).

**`stats` stays deferred; the metadata update landed in PR C-meta.** `GET …/work/stats` returns `workers_polling` — *"Number of workers that have polled in the last 30 seconds. Requires worker_id to be sent with poll requests."* — which needs poll-time `Anthropic-Worker-ID` tracking the current `Poll` does not record, so `stats` is the last slice-8 piece. The metadata update (`POST …/work/{work_id}`) is now implemented (see the PR C-meta section below).

### BYOC worker — the tool-exec driver over HTTP (slice 8, PR C2a)

The first half of the distributable worker (`internal/worker`): the **tool-exec driver**, the self_hosted twin of the platform executor's per-item processing. Given a session whose turn has suspended for built-in tool calls, `RunSessionTools` reads the session's outstanding `agent.tool_use` events over the wire, runs each in a local sandbox via the shared `toolset.Runner`, and posts a `user.tool_result` for each back through the session events API. Two deployment differences from the executor: the transport is **HTTP** (a BYOC worker has no database — it reaches the control plane only through the session API a `NewClient` SDK client authenticates to with the environment key as `Authorization: Bearer`), and the result event is **`user.tool_result`, not `agent.tool_result`** — so the control plane's own send-side state machine schedules the resume when a result completes the outstanding set, and the worker never enqueues a turn itself. The lease loop that polls the work queue, acks, heartbeats, and drives this — plus the `cmd/worker` binary and `traceparent` propagation — is PR C2b.

The driver mirrors the executor's proven semantics exactly: it diffs `agent.tool_use` against the answered set (an `agent.tool_result` **or** a `user.tool_result` both answer, keyed by `tool_use_id`), so a redundant reclaim re-runs nothing and provisions no sandbox; results post **per tool as each completes**, so a backend fault partway through leaves the tools that already ran answered on the log and a reclaim re-derives only the rest; a tool-level failure (missing file, nonzero exit) still posts an `is_error` result the model must see, and only a backend fault (sandbox gone) stops the set. Empty tool output posts **no content blocks** (stored as null content, which the brain's replay renders as a `tool_result` block with no `content` field — valid for the Messages API, where an empty text block is not), matching the executor's empty-content handling.

- **Event wire shapes are read from raw JSON, not the SDK's typed event union.** The union tracks the live API's tip and carries post-slice surface the worker has no need for; `unansweredToolUses` decodes only the three fields the diff needs (`id`, `name`, `input` on a use; `tool_use_id` on a result) from each event's raw bytes, so a schema drift in the checkout cannot break the worker. Writes still go through the SDK's typed `Send` (honoring the SDK as the wire transport), with the event's and text block's `type` discriminators set explicitly — the union marshaler does not fill them in, and the control plane requires them.
- **The control plane is a backstop, not the liveness gate.** A worker cannot forge an answer onto the append-only log: a result with a bad/duplicate `tool_use_id`, or one posted to an **archived** (read-only) session, is refused with a `400` the driver surfaces as an error rather than a silently-wedged session. But this is only a *partial* backstop — a post to a session that is merely **not running** (not archived) appends without resuming. The **complete** session-liveness gate — skip a session that is not running or is archived, and drain its work item — is the caller's, exactly as the executor puts it in its per-item orchestration (`sessionForRun`, under the session row lock) rather than its `runTools` core, which this driver is the analog of. It lands with the lease loop in **PR C2b**, which already loads the session to read the egress policy this driver takes as a parameter.
- **Known cost — full-history scan.** With no unanswered-only wire endpoint, `unansweredToolUses` pages the session's entire `agent.tool_use` and result history on every call to find the outstanding few — O(history) per tool batch, where the executor's local DB does it in one `EXISTS`. The outstanding set is always the last suspended turn's, so a newest-first early-stopping read could bound it later; the simple full scan is correct and is what this first cut ships.
- **The answering rule now lives in three places.** The "an `agent.tool_result` **or** `user.tool_result` answers an `agent.tool_use`" diff is expressed in `events.HasUnansweredToolUse` (canonical SQL), `executor.unansweredToolUses` (DB), and now `worker.unansweredToolUses` (wire) — each against a different data source, so they can't share a function. The result event types are the shared `domain` constants, and a code comment ties the three copies together so a new answering type is added to all.

### BYOC worker — the lease loop + `cmd/worker` binary (slice 8, PR C2b)

The runnable BYOC worker: `internal/worker.Worker` (the lease loop) plus the `cmd/worker` binary, the self_hosted twin of the platform executor's `Run`. `Worker.Run` polls the control plane's self_hosted work queue over HTTP, and for each item: acknowledges it, keeps a heartbeat alive while the C2a driver (`RunSessionTools`) runs the session's tools in a local Docker sandbox, and force-stops the item when the run ends. One session at a time, mirroring the reference `ant beta:worker` (whose loop lives in the SDK's `lib/environments`, not the thin CLI). Modelled field-for-field on the reference's behaviour:

- **Poll** sends `block_ms=999` and an `Anthropic-Worker-ID` header (auto-generated `<hostname>-<random>` when unset, as the reference does). Our control plane's poll is non-blocking (it does not yet honour `block_ms`), so an `EmptyPollSleep` paces empty polls client-side. A poll rejected for a bad key (401/403) is **fatal** — `Run` returns the error rather than spinning; any other poll error backs off and retries.
- **Ack then heartbeat-first.** The item is acked, then the heartbeat goroutine starts **before** the liveness check and the run — the reference's ordering, because every moment between the ack and the first heartbeat is a window the control plane sees no liveness signal in. The first beat sends `NO_HEARTBEAT` (claiming the lease, `starting → active`); each later beat echoes the server's prior `last_heartbeat` to extend it. When the control plane moves the item to `stopping`/`stopped`, declines to extend, or rejects the precondition (412 — another worker reclaimed it), the heartbeat cancels the run's context so an in-flight tool winds down. The cadence is derived from the response TTL (`ttl/2`, clamped `[1s, 30s]`); `desired_ttl_seconds` is not sent, per the reference.
- **The session-liveness gate lands here** (deferred from C2a). After ack, the worker `GET`s the session; a session that is not running or is archived is stale, so it runs nothing and lets the force-stop drain the item — the worker's equivalent of the executor's `sessionForRun` drain (the executor completes the item under the DB lock; the worker, with no DB, force-stops). This closes the gap where an archived session's `tool_exec` item (archiving does not touch the queue) would otherwise have its tools run once on customer compute before the post `400`s.
- **Graceful shutdown.** The binary wires a SIGINT/SIGTERM signal context; `Run` returns nil on cancellation. The force-stop runs on a fresh background context so the item is still stopped as the worker exits, and it ignores a `409` (already stopping/stopped), as the reference does.
- **The worker owns its sandbox shape.** `Image`/`Workdir`/`Networking` come from the worker's own config, not the wire: a self_hosted worker runs on the customer's compute and the session wire shape exposes it no per-session egress policy (only `status`/`archived_at`, which the liveness gate reads). A zero `Networking` gives the sandbox the default bridge network, and the customer's own infrastructure governs egress. This mirrors the executor, whose image/workdir are likewise a deployment choice.
- **`cmd/worker`** is configured entirely from the environment (`ANTHROPIC_BASE_URL`/`ANTHROPIC_ENVIRONMENT_ID`/`ANTHROPIC_ENVIRONMENT_KEY` required; `ANTHROPIC_WORKER_ID`/`WORKER_IMAGE`/`WORKER_WORKDIR`/`DOCKER_HOST`/`OTEL_*` optional). It has **no** `DATABASE_URL` — the worker reaches the control plane only over the wire, the point of BYOC.

PR C3 then landed the **dead-worker reclaim** of already-acked/heartbeating items (see the reclaim paragraph above): `Poll` reclaims a `starting`/`active` item whose heartbeat lease lapsed, and `Ack` now installs a startup lease so a `starting` item is reclaimed on a real lease rather than its short un-acked poll reservation. The worker sends its `Anthropic-Worker-ID` on every poll, but the control plane does not yet record it for `stats` (deferred).

### BYOC worker — `traceparent` propagation across the process boundary (slice 8, PR C2b-2)

Design principle 3 (observability built in): a session's model turns run on the platform brain and its tools run in a BYOC worker's own network, two processes — yet they should read as **one** OTel trace. This PR carries the turn's trace context from the enqueuing brain, through the control plane, to the worker's tool-execution span.

- **Capture at enqueue (brain).** When a turn suspends on a platform tool it enqueues the `tool_exec` item under the turn's `span.model_request` span; `queue.Enqueue` injects the active W3C trace context (`telemetry.Inject`) into a dedicated **`trace_context` `jsonb` column** on `work_items` (migration `0005`) — `NULL` when there is no active span. Only `tool_exec` captures it (the item a worker/executor *runs* as a tool); a `model_turn` drives the brain, which opens its own `model_request` span per turn and never reads this back, so populating it there would only persist an unread payload. The one wiring change in the brain: `settleTurn` now runs under the span-carrying context (`sctx`), not the plain `ctx`, so the enqueue inside `commitTurn`'s `Then` actually sees the turn's span.
- **Transport on the poll response (control plane).** `/work/poll` reads the picked item's `trace_context` and emits it as `traceparent`/`tracestate` **response headers**; the wire work body (`toWire`) never carries it. `pollWork` is therefore the one work endpoint that is a full `http.HandlerFunc` (it needs the `ResponseWriter` to set the header) rather than a typed handler.
- **Parent at run (worker).** `pollAck` reads the poll response via `option.WithResponseInto`, lifts the `traceparent`/`tracestate` headers into a carrier, and `handleItem` starts a `tool_exec` span parented on the extracted remote context (`telemetry.Extract`) — so the worker's tool run is a child of the brain turn that produced it. Only the tool run is spanned; the heartbeat is lease bookkeeping, outside the span.
- **Why a column, not `metadata`.** The plan sketched riding the work item's `metadata`, but that namespace is now client-updatable (PR C-meta), so an internal `traceparent` there would pollute the client-facing surface. A dedicated column keeps it out entirely — no leak, no strip step — and the header is emitted straight from it. Divergence noted with the work-API divergences above.

Storage is control-plane-internal; the transport (a response header the reference worker ignores) stays wire-compatible. The end-to-end proof is a worker test that drives a real enqueue-under-span → poll → run and asserts the recorded `tool_exec` span's parent is the enqueuing trace.

### work API — the work-item metadata update (slice 8, PR C-meta)

`POST /v1/environments/{id}/work/{work_id}` applies a client-managed metadata patch and returns the updated `BetaSelfHostedWork` (like ack/heartbeat/stop). The body is `{"metadata": {…}}` (required): a string value **upserts** a key, an explicit **null deletes** it, and an omitted key is preserved — the same patch semantics the wire documents and that session/agent metadata already use (null-deletes; an empty string here is a literal value, not a delete, unlike the environment-only empty-string rule). It closes the last worker-facing endpoint besides `stats`, and it is what makes the `metadata` namespace client-updatable — the very reason C2b-2 kept `traceparent` in its own column and out of `metadata`.

- **Atomic merge, not read-modify-write.** `queue.UpdateMetadata` persists the patch in one statement — `metadata = (metadata || $upserts::jsonb) − $deletes::text[]` — so a concurrent worker state transition on the same row cannot be clobbered and two overlapping patches cannot drop each other's writes. This diverges from the read-modify-write the versioned resources (session/agent/environment) use, and deliberately: a work item carries **no optimistic version**, and the wire update sends none, so there is no 409 to guard a read-modify-write with — the atomic SQL merge is the correct primitive. The nil-slice guards matter (a nil `deletes` would encode as SQL `NULL`, and `jsonb − NULL` is `NULL`, nulling the `NOT NULL` column; a nil `upserts` map would marshal to JSON `null`, and `jsonb || 'null'` makes an array — both are forced to empty).
- **Scoped and validated like the rest of the work API.** The patch reaches only a self_hosted `tool_exec` item in the caller's environment (`workAPIScope`); a `model_turn`, a cloud `tool_exec`, a wrong-env id, or an unknown id is `404`. The body goes through the shared `decodeObject`/`rejectUnknownKeys` path every other update endpoint uses (the 4 MiB cap and strict unknown-field rejection), and the string-or-null split is the shared `splitMetadataPatch` that also backs `patchMetadata`, so the work and session/agent patch semantics can't drift: `metadata` is required (`400` if absent — the wire marks it so and there is nothing else to patch), an unknown top-level field is `400`, and a non-string, non-null value is `400`. The patch is orthogonal to lifecycle state — any visible item is patchable in any state (queued through stopped).
- **Known gap — a `NUL` in a value is a `500`, not a `400`.** A metadata value containing `\u0000` is valid JSON but unstorable in Postgres `jsonb`, so it surfaces as a `500`. This is **pre-existing and codebase-wide**: every metadata endpoint (agents/environments/sessions create and update) writes the value into a `jsonb` column with no `NUL` guard and `500`s the same way — only the events-append path guards it (rejecting `\u0000` with a `400`). The right fix is that same guard hoisted into the shared metadata parse (`parseMetadata` + `splitMetadataPatch`) so every endpoint rejects it uniformly, deferred to its own change rather than diverging the work endpoint from its peers here.
- **Routing.** `POST .../work/{work_id}` now routes to the update; a `POST .../work/poll` therefore resolves as `work_id="poll"` and — with a valid patch body — `404`s on the nonexistent item, where it previously hit the method-less `405` fallback. An empty or malformed body is a `400` (metadata is required, validated before the item lookup), so the not-found outcome holds only for a valid patch. Other methods on `.../work/poll` (PUT/DELETE) still `405`. **Assumption to confirm against a real recording:** the `404` is inferred from the SDK's own route table — the reference exposes `POST v1/environments/{id}/work/{work_id}` (the `Update` method, verified `http.MethodPost` in `betaenvironmentwork.go`) and a separate `GET …/work/poll`, so a `POST` to the `poll` path folds into the `{work_id}` route here exactly as the SDK's client would address it. Whether the reference instead method-restricts the `poll` path to `GET` (answering `POST` with a `405`) is not decidable from the typed schema; no real client issues `POST …/work/poll`, and the pre-C-meta `405` was itself never verified for this edge.

### work API — the queue-stats endpoint (slice 8, PR C-stats)

`GET /v1/environments/{id}/work/stats` returns `BetaSelfHostedWorkQueueStats` — the last worker-facing work endpoint, closing slice 8. It is a **derived view over Postgres**, not a second source of truth: the SDK's field docs are Redis-consumer-group-native (`depth` = lag, `pending` = the PEL, `workers_polling` = the group's consumers), which all but confirms the reference queue is Redis Streams — but ours is the `work_items` table (the plan's `redis optional later`), so the figures are computed from it. **Divergence noted:** same wire numbers, Postgres provenance.

- **depth / pending partition the queued state.** The wire's "acknowledged" is our `Ack` (queued→starting), so an acked item has left the queue and counts toward neither. `depth` = queued items available to pick up (no reservation, or a poll reservation whose lease lapsed — the same `lease_expires_at < now()` boundary `Poll` re-offers on); `pending` = queued items polled but not yet acked (a live reservation). `oldest_queued_at` = the oldest queued item's `created_at` (depth + pending), null when the queue is empty. It is one aggregate `SELECT`, so an empty queue reports zeros rather than no row.
- **workers_polling needs poll-time tracking.** Migration `0006` adds `worker_polls (environment_id, worker_id, last_polled_at)`; `pollWork` reads the `Anthropic-Worker-ID` header and upserts the row — best-effort (a tracking failure never fails the poll, and a header-less poll is not attributed to any worker), at the cost of a second cheap round-trip before the poll returns. `workers_polling` counts distinct workers whose last poll landed within 30s, and its subquery carries the same self_hosted gate as `depth`/`pending`/`oldest`, so all four fields report on one queue. The same upsert reaps rows aged past that 30s window (excluding the current worker), so the table stays bounded by recently-active workers — without the reap, default worker ids being minted fresh per process would leak one permanent row per restart.
- **Scoped and routed like the rest of the work API.** Environment-key auth; the stats reach only the caller's self_hosted `tool_exec` queue (`workAPIScope`), so a cloud environment or a `model_turn` row never leaks in. `GET …/work/stats` is a literal segment that beats `GET …/work/{work_id}` (like `poll`); a non-GET on the path folds into the `{work_id}` routes (POST → 404, PUT/DELETE → 405).

## Kubernetes sandbox provider (slice 9)

`internal/sandbox/k8s` is the self-hosted twin of the Docker backend: one disposable Pod per session, driven over the Kubernetes API (`client-go`), passing the **same** `sandboxtest` contract suite the plan requires both backends to satisfy identically. Provision is get-or-create-then-wait-ready (a Pod schedules asynchronously, unlike a Docker container that is usable the moment create returns) and reclaims a Pod it created but could not bring to readiness — deleting only that exact Pod, guarded by its UID and a delete precondition and only while it is still not ready, on a detached context — so a retry starts clean rather than re-adopting a wedged one and without ever removing a same-named replacement or a Pod another executor has adopted; Destroy deletes with a zero grace period so the sandbox is final at the API-object level at once (a force delete does not block on the kubelet, so a partitioned node could lag, but the handle is dead either way); file I/O rides `exec` (no archive endpoint like Docker's), binary-safe over the stream — `ReadFile` rejecting symlinks and re-checking the size cap on the bytes read, `WriteFile` surfacing a failed write rather than reporting success. The Pod mounts **no ServiceAccount token**, so untrusted tool commands never inherit the namespace account's cluster credentials.

The hard part is the deadline, and it diverges from Docker by necessity. Docker reads a running exec's liveness and exit code out-of-band (`exec-inspect`); Kubernetes exposes no such handle — `StreamWithContext` couples the exit code to the exec stream, which a backgrounded straggler can hold open long after the command exited. So the in-Pod wrapper runs the command as a background child rather than via `exec`, records the command's pid to a file and, once it finishes, its exit code to another. Exec then keeps the docker backend's exact two-instant discipline — was it alive at the deadline, did it overrun the deadline-plus-slop — but answers each by a second `exec` that reads the pid file and `kill -0`s it, and reads the exit code from the file (the stream's straggler cannot delay either). The command runs under `setsid` (its own process group for the watchdog's group-kill, and no job-control "Killed" line leaking onto stderr), with its stderr handed a saved fd that both the command and the watchdog then close, so neither a straggler nor a sleeping watchdog pins the exec stream open and a quick timed command's stream EOFs at once instead of a poll interval late. The liveness probe treats a probe process killed out from under it as unknown (assume-alive), not as "dead", and the overrun verdict — the guarantee — is sticky, never retried, so no probe killed at the deadline can erase an overrun. All the crown-jewel adversarial cases (kill-the-watchdog, kill-the-watched-pid, overrun-then-exit-clean, mute-then-overrun, self-SIGKILL-is-not-a-timeout, straggler-is-not-the-command) pass against a real cluster. One axis is weaker than Docker by necessity: the pid the verdict reads lives in the sandbox's own filesystem (Kubernetes has no out-of-band exec handle), so a *malicious* command that both kills its watchdog and overwrites that pid file can hide an overrun — the same malicious-tenant case the derived-name adoption check leaves out of scope in single-tenant v1; the honest runaway the deadline exists for forges nothing.

`limited` networking fails closed like Docker's `NetworkMode: none`: an init container (a tiny utility image with an `ip`, since the sandbox image is only guaranteed `/bin/bash`; NET_ADMIN) flushes the Pod netns's routing table before the sandbox container starts, then re-reads it and exits non-zero if any IPv4 route survived — so a flush that silently no-ops (a busybox `ip` missing the selector, an ineffective NET_ADMIN) fails the Pod closed rather than starting a "limited" sandbox with egress. It is not equivalent to `NetworkMode: none` on every cluster: it does not remove the interface (raw AF_PACKET sockets can still reach the segment) and inspects only the main IPv4 table, so a policy-routing CNI or dual-stack IPv6 needs the reserved egress proxy for a complete cutoff. The contract suite runs against a **kind** cluster (a missing cluster is a hard failure, not a skip, mirroring the Docker daemon rule); CI provisions kind before the coverage run. Error branches a live cluster cannot easily stage (adoption, foreign-pod rejection, create/delete faults, not-found reclassification, a created-but-unready Pod reclaimed) are covered by fake-clientset unit tests.

**Backend selection and the Helm chart are both wired.** `cmd/executor` and `cmd/worker` build their sandbox provider through `internal/sandbox/backend`, which chooses Docker or Kubernetes from `SANDBOX_BACKEND` (empty defaults to `docker`, so an existing deployment is unchanged) plus that backend's own env vars (`DOCKER_HOST`, or `SANDBOX_K8S_KUBECONFIG`/`_CONTEXT`/`_NAMESPACE`/`_NETSETUP_IMAGE`). The **Helm chart** (`deploy/helm/managed-agent-platform`) deploys the control plane (Deployment + Service), the brain, and the executor (each independently scalable), with the executor wired to the `k8s` sandbox backend over in-cluster config and granted a namespaced Role for exactly the pod-lifecycle + `pods/exec` verbs the provider calls; an optional in-cluster Postgres (StatefulSet — toggle off and point `externalDatabase.url` at a managed database) and OTLP endpoint values complete it. The BYOC worker is intentionally out of the chart — it runs on the customer's own compute. Two deliberate divergences from the plan's chart sketch: Postgres ships **inline rather than as a subchart** (an external chart-repo dependency undercuts air-gap self-hosting; production runs its own database anyway), and the optional **gVisor `RuntimeClass` is deferred** until the K8s provider sets `runtimeClassName` on sandbox pods (shipping it now would be unwired). Container images are operator-supplied — this repo publishes none yet, so the chart references `{registry}/{repository}/{component}:{tag}` and is validated by `helm lint`, `helm template` across the internal-Postgres / external-database / bring-your-own-Secret paths, and a **server-side `kubectl apply --dry-run` against a kind cluster** (all ten objects schema-valid); a real end-to-end `helm install` awaits published images. A `helm` CI job lints and renders the chart across the internal-Postgres, external-database, and existing-Secret paths, asserts the rendered brain model-providers file is the JSON array `LoadRoutes` requires (a shape mismatch would crash-loop the brain at deploy time — the one bug the verifier caught pre-merge), and checks a required-value guard fails.

**Local development stack (docker compose).** The compose companion to the Helm chart, filling the previously-empty `deploy/compose/` the CLAUDE.md dev commands referenced. A repo-root multi-stage `Dockerfile` builds all four binaries (`CGO_ENABLED=0 go build ./cmd/...`) into one `debian:stable-slim` image, with the binaries at the **filesystem root** (`/controlplane` …) so the same image also satisfies the Helm chart's `command: ["/controlplane"]` — one image for both deploy paths. `deploy/compose/docker-compose.yml` runs controlplane + brain + executor (one shared image, selected per service by an absolute-path `command`) against a bundled `postgres:16-alpine`, plus an optional Jaeger behind an `observability` profile. The control-plane port binds **loopback by default** (the committed API key is a placeholder; `CONTROLPLANE_BIND=0.0.0.0` with a real key exposes it), and the brain's model-routing mount **defaults to the committed example** (`MODEL_PROVIDERS_FILE` points it at a real config), so a bare `docker compose up` comes up cleanly with no missing-file footgun. A YAML anchor shares the build/env/`depends_on` across the three app services; they gate on Postgres's `pg_isready` healthcheck and each auto-applies migrations on connect (advisory-locked — the executor and brain talk to Postgres directly, not the controlplane, so no inter-app ordering is needed). The executor runs the **docker** sandbox backend over the mounted host `/var/run/docker.sock` (local-dev convenience; the k8s backend is the production path). Model routing (`model-providers.json`, a JSON array) and secrets (`CONTROLPLANE_API_KEY`, Postgres password) come from gitignored files with committed `.example` templates; the BYOC worker is excluded (customer compute). **Verified end to end on Docker 29 / Compose v5:** the image builds, all four services start clean, migrations apply, and the control plane serves the API — authenticated list `200` (`{"data":[],"next_page":null}`), missing key `401`, unknown-field validation `400` with a `request_id`. This is the local stack the slice-8 `ant beta:worker` acceptance ran against (now passed — see the acceptance record above).

## Eval test system (phase 1, #30)

The first test that drives a whole session the way a customer does — public REST → brain → a real model → work queue → executor → Docker sandbox → SSE idle — and grades the transcript deterministically. Every other loop test in the repo scripts the provider; nothing before this exercised the real product path, and #30 also flagged that `.env` presence alone was triggering a paid model call during an ordinary `go test`.

**Harness form.** A top-level `evals/` package of `*_test.go` only — no runner binary, because `go test` already gives subtests, timeouts, `-v`, and panic-safe cleanup. `TestEvals` composes the platform in one process exactly as `cmd/*` do: a `pgtest` Postgres, the real `api.NewHandler`, a `provider.Registry` routing `*` to the `.env` endpoint, and live `brain` and `executor` loops against `docker.New`. Only `main()` glue is bypassed (CI's compose job smokes that). A hand-rolled REST client speaks `map[string]any`, never the domain structs, so a wire regression a struct tag would round-trip past stays visible to a grader. The suite is opt-in through `RUN_EVALS`, a second gated tier in `internal/modeltest` alongside the provider smokes' `RUN_LIVE_MODEL_TESTS`: unset skips (an ordinary `go test ./...` makes zero paid calls even with `.env` present — the behaviour change #30 demanded), set-but-misconfigured fails rather than skips. A `TierEnabled` answers the one caller a `*testing.T` skip cannot serve — `TestMain`, which starts Postgres before any test can skip.

**Grading is deterministic and code-based**, never an LLM judge — a judge's own drift is indistinguishable from the drift the suite exists to catch. Each prompt demands a per-trial random nonce, so an exact-match check tests the agent rather than the grader's generosity. Every trial runs a core pack (reaches idle with `stop_reason.type == end_turn`; no `session.error`; every `agent.tool_use` joined by exactly one `agent.tool_result`; token usage populated; the idle observed on the SSE stream), and each finding is classed **P**latform (our bug, a red run to fix), **M**odel (the model wandered), or **E**ither, so a red run says whose problem it is rather than "probably the model". Artifacts land in `evals/artifacts/`: a `report.json`, a `summary.md`, and one transcript per failed trial — including an aborted one, fetched best-effort in the deferred recorder because a drive timeout is the failure triage most needs. The report reduces the endpoint to host:port and scrubs known secrets from every rendered artifact, because a credential in `MODEL_BASE_URL`'s query would otherwise ride a transport error's quoted URL onto disk.

**The ten tasks** exercise the built-in toolset — bash, read, edit and grep are graded by a result contract tying a call to its own output, and glob and write by a required tool-use floor with the artifact checked separately (a bare path list and a written file carry no output to pin, and the prompts name both tools) — single and multi turn, allow and deny, seeded and unseeded, with three negatives. Tasks 1–3 (`fib-quickstart`, `echo-notool`, `shell-state`) landed with the harness; tasks 4–10 added three mechanisms. **Seed planting** writes files into the session's container before turn 1 by pre-provisioning it; the executor adopts that same container (by session label) when it runs the first tool, so the agent sees the seeds — used by `edit-config`, `needle-search`, `perm-deny`, `view-range`. **Gated toolsets** (`always_ask` via `default_config`) and a **confirmation-aware drive loop** power the permission pair: `perm-allow` and `perm-deny` drive the bridge end to end — a gated tool suspends the session on a `requires_action` idle, the loop posts a `user.tool_confirmation` (allow or deny) referencing the event id `requires_action` named, and grading pins the pause, the `evaluated_permission == "ask"` stamp, the result sequenced after the approval, and — on deny — the synthesized `is_error` result carrying the deny message with the seeded file left untouched. `exit-code` pins a failed command's `exit code:` trailer (which exists only in the real tool result, so reporting it proves the model consumed it); `journal-multiturn` pins event replay and sandbox reuse across two turns; `view-range` pins `read`'s `view_range` slicing byte-for-byte as an off-by-one guard.

Prompts are written the way the docs tell a user to write them — a prompt tuned until only our platform's quirks satisfy it stops being a regression test. Two that a refusal-prone live model (MiniMax-M3) balked at were reworded to exercise the platform rather than trip a safety reflex: `perm-deny`'s "delete a file in a `protected` directory" became a benign append the reviewer declines, and `view-range`'s "SECRET" marker copied "to another file" — which the model read as exfiltration — became a plain marker copied to a file. All ten run 10/10 green live.

Deliberately deferred and filed as issues: a daily scheduled CI run ([#96](https://github.com/OpenSDLC-Dev/managed-agent-platform/issues/96), needs repo `MODEL_*` secrets), `tool_choice`/`disable_parallel_tool_use` for phase 1.5 (on #30), and production sandbox reaping ([#64](https://github.com/OpenSDLC-Dev/managed-agent-platform/issues/64)). The harness reaps its own containers at stack teardown.
